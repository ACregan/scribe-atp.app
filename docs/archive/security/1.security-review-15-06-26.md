## Overview

This review was conducted as the first formal security pass against the Scribe ATP codebase and its VPS infrastructure. The scope covered:

- All server-side route actions and loaders (`app/routes/`)
- Authentication and session handling (`app/services/auth.server.ts`)
- HTML sanitisation of user-generated and AT-Protocol-sourced content
- HTTP response headers (both application-level and nginx)
- Client-side resource loading (Google Fonts, inline scripts)
- Image Service Express app (`image-service/`)
- nginx virtual host configuration (`/etc/nginx/sites-enabled/scribe-atp.app`)
- VPS process user and filesystem layout
- Audit logging coverage

Twelve findings were recorded: four Critical/High, five Medium, three Low. All have been addressed or explicitly deferred with a documented plan.

## Findings

### 1. Missing HTTP Security Headers

**OWASP Category:** A05 — Security Misconfiguration
**Severity:** High
**Status:** Mitigated

**Description**

The application returned no security-hardening HTTP headers. An attacker could exploit the absence of:

- `X-Frame-Options` — enabling clickjacking attacks (embedding the app in a hostile iframe)
- `X-Content-Type-Options` — enabling MIME-type sniffing, which can turn an uploaded file into an executable script
- `Referrer-Policy` — leaking session information in the `Referer` header to third-party domains
- `Strict-Transport-Security` — allowing protocol downgrade from HTTPS to HTTP
- `Permissions-Policy` — leaving device APIs (camera, microphone, geolocation) unconstrained

**Mitigation**

A dedicated `applySecurityHeaders()` function was added to `app/entry.server.tsx`. It is called once at the top of the request handler and sets all five headers on every response:

```ts
function applySecurityHeaders(headers: Headers): void {
  headers.set("X-Frame-Options", "SAMEORIGIN");
  headers.set("X-Content-Type-Options", "nosniff");
  headers.set("Referrer-Policy", "strict-origin-when-cross-origin");
  headers.set("Permissions-Policy", "camera=(), microphone=(), geolocation=()");
  headers.set("Content-Security-Policy", CSP); // see Finding 11
  if (isProduction) {
    headers.set(
      "Strict-Transport-Security",
      "max-age=31536000; includeSubDomains",
    );
  }
}
```

HSTS is production-only to avoid breaking the dev server (which runs over HTTP). SAMEORIGIN was chosen for `X-Frame-Options` rather than `DENY` to allow any future dashboard embedding from the same origin.

**Files changed:** `app/entry.server.tsx`

### 2. Unvalidated Image URL Protocols

**OWASP Category:** A03 — Injection
**Severity:** Medium
**Status:** Mitigated

**Description**

The article create/edit forms and the site configure form accepted image URL inputs (`splashImageUrl`, `logoImageUrl`) without validating that they used an HTTPS scheme. This could allow `javascript:` URLs or `data:` URIs to reach rendered `<img src>` attributes, creating a potential XSS vector in consumer sites that render the article HTML.

**Mitigation**

A shared regex constant `IMAGE_URL_RE = /^https:\/\//i` was added to `app/constants.ts`. All three routes (`create.tsx`, `edit.tsx`, `configure.tsx`) now validate image URL fields against this regex server-side in their actions and return a form error on failure. HTTP-only URLs are also rejected — HTTPS is required.

```ts
if (splashImageUrl && !IMAGE_URL_RE.test(splashImageUrl))
  return { error: "Splash Image URL must start with https://." };
```

**Files changed:** `app/constants.ts`, `app/routes/article/create/create.tsx`, `app/routes/article/edit/edit.tsx`, `app/routes/site/configure/configure.tsx`

### 3. Unsanitised HTML Rendering

**OWASP Category:** A03 — Injection (Stored XSS)
**Severity:** High
**Status:** Mitigated

**Description**

Article content is stored as HTML produced by the Lexical rich text editor. The article view route (`/article/view/:articleUrl`) rendered this HTML using React's `dangerouslySetInnerHTML` without any sanitisation pass. Since article content can also be fetched directly from the AT Protocol by external consumers, injecting malicious HTML into a PDS record and triggering a view via the Scribe URL would cause the stored XSS to execute in the viewer's browser.

**Mitigation**

`isomorphic-dompurify` was added as a dependency and applied server-side in the article view loader before the HTML reaches the component:

```ts
import DOMPurify from "isomorphic-dompurify";

// in loader:
const safeContent = DOMPurify.sanitize(rawContent);
return { article: { ...article, content: safeContent } };
```

`isomorphic-dompurify` supports both browser and Node.js environments (using `jsdom` on the server) so the same import works in SSR and client contexts.

**Files changed:** `app/routes/article/view/view.tsx`
**Dependency added:** `isomorphic-dompurify`

### 4. E2E Bypass Exploitable in Production

**OWASP Category:** A04 — Insecure Design
**Severity:** Medium
**Status:** Mitigated

**Description**

The `useRealOAuth` flag in `app/services/auth.server.ts` evaluates to `false` when `E2E=true` is set in the environment, even in a production build. Setting `E2E=true` bypasses Bluesky OAuth entirely — any handle submitted on the login form would be accepted as authenticated with no credentials checked. The E2E escape hatch exists for CI (where `react-router-serve` sets `NODE_ENV=production` but real OAuth is impossible), but if `E2E=true` accidentally reached a live server environment, the entire authentication system would silently disable itself.

**Mitigation**

A startup guard was added immediately after the `SESSION_SECRET` check in `auth.server.ts`:

```ts
if (isProduction && process.env.E2E === "true" && process.env.CI !== "true") {
  throw new Error("E2E=true must not be set in production outside of CI");
}
```

The guard requires `CI=true` to coexist with `E2E=true` in production — this allows GitLab CI (which sets `CI=true` automatically) to continue working while blocking accidental or malicious use of the bypass flag on a live server.

**Files changed:** `app/services/auth.server.ts`

### 5. Login Endpoint Not Rate-Limited

**OWASP Category:** A04 — Insecure Design / A07 — Identification and Authentication Failures
**Severity:** Medium
**Status:** Mitigated

**Description**

The `/login` route action had no per-IP throttle. An attacker could submit an unlimited number of handle guesses, enumerating valid Bluesky handles by observing whether the app forwarded the request to Bluesky's PAR endpoint (valid handle) or returned an immediate error (invalid handle). Bluesky's own servers have rate limits, but the local endpoint was unbounded and could also be used for handle enumeration without triggering Bluesky's limits.

**Mitigation**

A `login_attempts` table was added to the SQLite database (`data/oauth.db`) tracking `(ip, created_at)` per attempt. The `loginAttempts` export from `db.server.ts` provides `record(ip)`, `count(ip)`, and `isLimited(ip)` methods. The limit is 10 attempts within a rolling 15-minute window per IP address.

The login action extracts the client IP from `X-Forwarded-For` (set by nginx) and checks the limit before forwarding the request:

```ts
const ip =
  request.headers.get("X-Forwarded-For")?.split(",")[0].trim() ?? "unknown";
if (loginAttempts.isLimited(ip)) {
  return { error: "Too many login attempts. Please try again in 15 minutes." };
}
loginAttempts.record(ip);
```

Stale attempts (older than 15 minutes) are pruned on application startup.

**HMR note:** The migration that creates `login_attempts` was initially gated behind the `global.__db` singleton check, which caused a `no such table: login_attempts` error on HMR reloads (the table was created on first boot but the migration was skipped on subsequent HMR reconnects). The fix was to always run `migrate()` outside the singleton guard — all migrations use `IF NOT EXISTS` so they are idempotent.

**Files changed:** `app/services/db.server.ts`, `app/routes/login/login.tsx`

### 6. Google Fonts External Dependency

**OWASP Category:** A08 — Software and Data Integrity Failures
**Severity:** Low
**Status:** Mitigated

**Description**

The application loaded the Inter font family from Google Fonts via two preconnect links and a stylesheet link to `fonts.googleapis.com`. This created three concerns:

1. **Supply chain risk** — if `fonts.googleapis.com` or `fonts.gstatic.com` were compromised, malicious CSS or fonts could be served to all users
2. **Privacy** — every page load issued a request to Google servers, leaking user IP addresses and visit metadata
3. **CSP complexity** — adding `font-src` and `style-src` entries for Google Fonts would have widened the Content Security Policy significantly (see Finding 11)
4. **Reliability** — external dependency creates a single point of failure for typography

The broken `@font-face` declarations that already existed in `app/styles/typography.css` pointed to paths that didn't resolve (`../../fonts/Inter/`), confirming the Inter woff2 files were already present in `public/fonts/Inter/` but not being used correctly.

**Mitigation**

The three Google Fonts link tags were removed from `app/root.tsx` and replaced with a single local stylesheet reference:

```ts
{ rel: "stylesheet", href: "/fonts/Inter/inter.css" }
```

The broken `@font-face` blocks in `typography.css` were removed. The `inter.css` file already present at `public/fonts/Inter/inter.css` contains correct relative-URL `@font-face` declarations for all weights. No font files needed to be downloaded — they were already present in the repo.

This change also eliminated the need for `fonts.googleapis.com` and `fonts.gstatic.com` in the CSP (see Finding 11).

**Files changed:** `app/root.tsx`, `app/styles/typography.css`

### 7. nginx Access Logging Disabled on Main App Location

**OWASP Category:** A05 — Security Misconfiguration / A09 — Security Logging and Monitoring Failures
**Severity:** High
**Status:** Mitigated (server configuration)

**Description**

The nginx virtual host configuration for `scribe-atp.app` contained `access_log off;` in the `location /` block, meaning HTTP request logs for the main application were silently discarded. In the event of a security incident, there would be no HTTP-level audit trail: no record of which IPs accessed the app, which routes were hit, response status codes, or timing.

**Mitigation**

The `access_log off;` directive was removed from the `location /` block in the nginx configuration on the VPS (`/etc/nginx/sites-enabled/scribe-atp.app`). Access logs now flow to the default nginx access log location.

The `docs/nginx/image-library.conf` reference file was updated to reflect this change.

**Infrastructure change (VPS only) — no application code change**

### 8. Node.js Processes Running as Root

**OWASP Category:** A05 — Security Misconfiguration
**Severity:** High
**Status:** Partially mitigated — full migration deferred

**Description**

Investigation confirmed that both the main React Router app and the Image Service Express process run as `root` on the VPS. Nine sites across the server share this configuration. If either Node.js process were compromised (via a dependency vulnerability or application bug), the attacker would immediately have full root privileges — no privilege escalation step required.

The image storage path (`/root/server/SITES/scribe-atp.app/image-storage/`) being inside `/root/` confirmed the root-user origin. Non-root processes cannot traverse `/root/` (mode 700), so migrating to a service user would require relocating all site files.

**Mitigation (partial)**

Image storage was moved from `/root/server/SITES/scribe-atp.app/image-storage/` to `/var/scribe/images/` (101/101 images verified). The nginx alias directive in `docs/nginx/image-library.conf` was updated. The `IMAGE_STORAGE_ROOT` env var in the PM2 config was updated to `/var/scribe/images`. The Image Service was restarted to pick up the new path.

**Deferred:** Full migration of all nine sites to a dedicated service user (e.g. `scribe`) requires a planned maintenance window. The new user would need read access to all site SSL certs, write access to the new storage paths, and all nine site processes would need to be stopped, relocated out of `/root/`, and restarted. This is tracked as a future maintenance task.

**Files changed:** `docs/nginx/image-library.conf`
**VPS changes:** nginx alias updated, `IMAGE_STORAGE_ROOT` env updated, Image Service restarted

### 9. No Structured Audit Logging

**OWASP Category:** A09 — Security Logging and Monitoring Failures
**Severity:** Low
**Status:** Mitigated

**Description**

All server output was via `console.error` with no structure. There was no audit trail for security-relevant events: login attempts, article creation/deletion, site operations, image uploads, or the developer nuke tool. In the event of an incident, it would be impossible to reconstruct what actions were performed and by whom.

**Mitigation**

`pino` was added as a structured JSON logging library. A shared logger module (`shared/logger.ts`) creates a single pino instance with `name: "scribe-atp"`, re-exported via `app/services/logger.server.ts` for app routes and imported directly by the image service.

The following security events are now emitted as structured JSON to stdout (captured by PM2):

| Event                | Fields                                              | Emitter                            |
| -------------------- | --------------------------------------------------- | ---------------------------------- |
| `auth.login_attempt` | ip, handle, outcome (`initiated`/`blocked`/`error`) | `login.tsx`                        |
| `article.create`     | user_did, rkey, site_count                          | `article.server.ts`                |
| `article.update`     | user_did, rkey, old_rkey?, slug_renamed             | `article.server.ts`                |
| `article.delete`     | user_did, rkey                                      | `list.tsx`                         |
| `site.create`        | user_did, rkey                                      | `sites.tsx`                        |
| `site.delete`        | user_did, rkey                                      | `sites.tsx`                        |
| `site.configure`     | user_did, rkey                                      | `configure.tsx`                    |
| `article.nuke`       | user_did, collection_count                          | `home.tsx` (warn level)            |
| `image.upload`       | user_did, uuid, original_name                       | `image-service/src/queue.ts`       |
| `image.delete`       | user_did, image_id, filename                        | `image-service/src/deleteImage.ts` |

All events include an ISO 8601 timestamp and log level from pino. The `article.nuke` event is logged at `warn` level as it is a destructive bulk operation.

**Files created:** `shared/logger.ts`, `app/services/logger.server.ts`
**Files changed:** `app/routes/login/login.tsx`, `app/services/article.server.ts`, `app/routes/article/list/list.tsx`, `app/routes/sites/sites.tsx`, `app/routes/site/configure/configure.tsx`, `app/routes/home/home.tsx`, `image-service/src/queue.ts`, `image-service/src/deleteImage.ts`
**Dependency added:** `pino`

### 10. No Content Security Policy

**OWASP Category:** A05 — Security Misconfiguration
**Severity:** Medium
**Status:** Mitigated

**Description**

No `Content-Security-Policy` header was set. Without a CSP, a successful XSS injection (e.g. via article content from a compromised AT Protocol record) has no browser-enforced restriction on what it can do: load arbitrary scripts, exfiltrate data, or perform actions on behalf of the user.

Finding 6 (self-hosting Inter) was intentionally implemented before this finding to avoid needing `fonts.googleapis.com` in the `font-src` and `style-src` directives, simplifying the policy considerably.

**Inline script challenge**

The application uses one inline script in the `<head>` — the theme detection script that reads `prefers-color-scheme` before paint to prevent a light/dark flash on first visit. This script is static (content never changes at runtime). Inline scripts are blocked by CSP by default; two options exist:

1. **Per-request nonce** — inject a random `nonce` attribute on the script tag each request, add `'nonce-{value}'` to `script-src`. More complex (requires nonce threading through SSR), and incompatible with cached responses.
2. **SHA-256 hash** — compute the hash of the static script once, hardcode it in `script-src`. Simple and cache-friendly.

The SHA-256 approach was chosen. The hash was computed with:

```bash
node -e "
  const c = require('crypto');
  const s = \"(function(){try{if(!document.cookie.match(/(?:^|;\\s*)theme=/)){var dark=window.matchMedia('(prefers-color-scheme: dark)').matches;if(dark){document.documentElement.setAttribute('data-theme','dark');}}}catch(e){}})();\";
  console.log(c.createHash('sha256').update(s,'utf8').digest('base64'));
"
// → LY0WmDNgT2tT94lZyQ2tYxwi9P0ohZ5I8xEYNyKfi7Q=
```

**`style-src: 'unsafe-inline'`**

The Lexical rich text editor applies text formatting (colour, font size, background) via inline `style` attributes on spans. Article HTML from `dangerouslySetInnerHTML` may also contain inline styles. There is no practical way to move these to external stylesheets, so `'unsafe-inline'` for `style-src` is required. This is a known limitation — the XSS risk from inline styles is significantly lower than from inline scripts.

**Final CSP**

```
default-src 'self';
script-src 'self' 'sha256-LY0WmDNgT2tT94lZyQ2tYxwi9P0ohZ5I8xEYNyKfi7Q=';
style-src 'self' 'unsafe-inline';
img-src 'self' data: https:;
font-src 'self';
connect-src 'self';
object-src 'none';
base-uri 'self';
form-action 'self';
frame-ancestors 'self'
```

- `img-src https:` — article HTML may embed images from external HTTPS URLs
- `connect-src 'self'` — covers image upload XHR and SSE progress stream
- `object-src 'none'` — blocks Flash and other plugin content
- `base-uri 'self'` — prevents `<base href>` injection from redirecting relative URLs to an attacker's domain
- `form-action 'self'` — prevents form hijacking via `action` attribute injection
- `frame-ancestors 'self'` — redundant with `X-Frame-Options: SAMEORIGIN` but provides defence in depth for browsers that support CSP but not X-Frame-Options

The hash is documented in `entry.server.tsx` with a comment explaining how to recompute it if the script content ever changes.

**Files changed:** `app/entry.server.tsx`

## Infrastructure Notes

### nginx `add_header` inheritance

nginx `add_header` directives in a parent `server {}` block are **not inherited** by child `location {}` blocks that define their own `add_header` directives. This is a well-known nginx inheritance quirk. The Scribe nginx config was audited and security headers were confirmed to be set correctly — no silent gaps.

### SESSION_SECRET handling

The `SESSION_SECRET` value is managed via PM2 config (`startup.pm2.config.js`) on the VPS. This file should not be committed to the repository and should be kept outside web-accessible paths. If the secret is ever exposed (e.g. by pasting a config into a conversation or issue), it should be rotated immediately:

```bash
node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"
```
