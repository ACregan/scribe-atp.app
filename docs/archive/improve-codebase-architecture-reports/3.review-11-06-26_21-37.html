ArticleRef Propagation Tax

11 sites must change in lockstep on every new Article field

Strong

Files involved

app/hooks/types.ts app/services/article.server.ts app/routes/article/site-list/siteTree.ts app/routes/article/create/create.tsx app/routes/article/edit/edit.tsx app/services/articleSiteSync.server.ts

Problem

ArticleRef is a cached snapshot of Article metadata stored inside Site records. When any new field is added to app.scribe.article, the same field must be manually added in 11 places — including four separate field maps inside siteTree.ts alone (two in buildTreeFromSite, two in treeToSiteData). The CLAUDE.md already flags this as a known cost and asks developers to remember to update all propagation sites. That's a documentation-as-guardrail, not an architecture-as-guardrail.

Benefits

  • Locality: one place to add a field instead of eleven. The four duplicate maps in siteTree become two clearly-named conversion functions.

  • Tests: the existing round-trip tests still pass with minimal changes; new field tests become trivially writable against the conversion functions.

755 lines, 7 state variables, zero unit tests

Strong

Problem

The route component owns: DnD sensor setup, drag state, dirty tracking, unsaved-changes blocker, group-state reconciliation after loader revalidation, create-group modal (with its own slug autofill), unsaved-changes modal, save/delete fetchers, and all render logic. This is seven distinct concerns in one file.

The fragile area is the knownGroupSlugsRef + savedTreeRef dance: when the loader revalidates after a group is created, the component must detect new slugs and append them to both tree and savedTreeRef simultaneously — otherwise the new group appears as an unsaved change. This is correct today but has no unit test harness. A future maintainer adding another state update in this area has no safety net.

Before — everything in site-list.tsx

site-list.tsx (755 lines)
📦 loader (auth + PDS fetch)
⚡ action (createGroup, deleteGroup, saveSite, removeArticle)
🎯 DnD sensor setup + drag event handlers
💾 dirty tracking + savedTreeRef
🚧 unsaved-changes blocker + proceed-after-save ref
🔄 knownGroupSlugs reconciliation with loader
🪟 CreateGroupModal (slug autofill)
🪟 UnsavedChangesModal
🖼 full render tree

After — extracted seams

useSiteListDnD(tree, setTree)
DnD sensors, handlers, activeItem state. Pure hook, mockable.
useDirtyTree(initialTree)
tree + savedTreeRef + isDirty + sync logic. The fragile dance lives here, testable in isolation.
CreateGroupModal (app/components/)
Reusable — already needed in groups.tsx. Slug autofill shared.
UnsavedChangesModal (app/components/)
Generic — could be used anywhere useBlocker() appears.
site-list.tsx (~280 lines)
Loader, action, wiring. Readable.

Benefits

  • Locality: the fragile savedTreeRef/knownGroupSlugs logic lives in one testable hook. A bug there doesn't require reading 700 lines of context to find.
  • Leverage: CreateGroupModal is already duplicated (see candidate 3); extracting it kills two birds. UnsavedChangesModal appears in create.tsx and edit.tsx too — same payoff.
  • Tests: useDirtyTree can be unit-tested with renderHook; the fragile group-reconciliation race condition becomes a pure state test.

create.tsx / edit.tsx Shared Logic Spread

~84 lines duplicated; hasTextContent copied verbatim; site-selection state reinvented

Worth exploring

Files involved

app/routes/article/create/create.tsx app/routes/article/edit/edit.tsx app/services/article.server.ts app/components/ArticleForm/ArticleForm.tsx

Problem

Both routes share the same conceptual lifecycle: load sites for the multi-select, handle site assignment changes, validate fields, build the PDS record, show a dirty-state warning, toast on success, and navigate. They've been extracted at the data layer (article.server.ts) and at the render layer (ArticleForm), but not at the state layer — the glue logic between them is duplicated.

Duplicated exactly

hasTextContent(html) — strips tags, checks non-empty. Identical in both files.
selectedSites: string[] + setSelectedSites — same state shape, same initialisation from loader, same change handler
Toast-then-navigate pattern on action success — same useEffect shape

Nearly identical but intentionally different

Unsaved-changes blocker — same useBlocker pattern; create checks isDirty from form fields; edit checks isDirty from CID mismatch or site changes
Slug autofill — create has title→slug sync; edit never autofills (intentional; slug is the rkey)

Solution

Move hasTextContent into article.server.ts (it's a pure utility used server-side in both actions). Extract a useArticleSites(initialSiteRkeys) hook that owns selectedSites state and the add/remove change handler — the hook's interface is the same in both routes, even if the initialisation differs slightly. The toast-then-navigate effect is already consistent; extracting it further would be over-engineering.

Benefits

  • Locality: hasTextContent lives next to the other article validation logic rather than silently diverging across two files.
  • Leverage: modest — the duplication isn't causing bugs yet. Worth doing alongside a wider tidy of the article routes.

toSlug Duplication

Defined independently in siteTree.ts and create.tsx; groups.tsx already imports correctly

Worth exploring

Files involved

  return title.toLowerCase()
    .replace(/\s+/g, '-')

}

function slugify(title: string) {

}

groups.tsx (correct import)

Solution

Dual SiteData Types

Two types share the same name in different layers; one is a display card, one is a tree manifest

Worth exploring

Files involved

app/components/types.ts app/routes/article/site-list/siteTree.ts

components/types.ts — display shape

type SiteData = {
  rkey, cid, title, url, urlPrefix,
  description?, splashImageUrl?,
  logoImageUrl?,
  groupCount,   // ← summary count
  articleCount  // ← summary count
}

Used by: SiteTile, SiteListItem, sites.tsx, list.tsx

siteTree.ts — manifest shape

type SiteData = {
  rkey, cid, url, title, urlPrefix,
  groups: SiteGroup[],       // ← full array
  ungroupedArticles: ArticleRef[] // ← full array
}

Used by: site-list.tsx loader, devFixtures

Solution

The display shape is a Site summary card — rename it to SiteCard in components/types.ts. The manifest shape is the full Site manifest used for DnD tree construction — rename it to SiteManifest in siteTree.ts. Both names map directly to domain concepts. The UBIQUITOUS_LANGUAGE glossary doesn't have a term for either shape, so this is a naming addition, not a conflict. Update callers (SiteTile, SiteListItem, etc.) to import SiteCard.

Note

CLAUDE.md already acknowledges this split exists and calls the shapes intentionally separate. This is a pure rename for navigational clarity, not a structural change. Low risk, low friction.

article.server.ts — Too Shallow to Justify?

74 lines, 2 callers, 4 micro-transforms; narrow extraction that may not earn its interface

Speculative

Files involved

app/services/article.server.ts app/routes/article/create/create.tsx app/routes/article/edit/edit.tsx

Deletion test

Delete article.server.ts → validateArticleFields (3 lines), buildArticleRecord (5 lines), and buildArticleRef (7 lines) each land in two places. That's a small duplication but buildArticleRef is load-bearing for Candidate 1 — it's the one canonical construction site. Don't delete. But consider whether the module should grow rather than stay at 74 lines.

Alternative

Rather than deleting or leaving as-is, article.server.ts could absorb hasTextContent (from Candidate 3), addArticleToSites is already here but not listed in CLAUDE.md's exports table — it's the right home for site-assignment logic. Make the module deeper by drawing more article-lifecycle logic into it, rather than questioning its existence.

Top recommendation

Start with Candidate 1 — ArticleRef Propagation. It's the only architectural issue that will get worse with every feature you add. The fix is contained and surgical: replace the four duplicate maps in siteTree.ts with two named conversion functions (articleRefFromNode / nodeFromArticleRef), and update the round-trip tests. Nothing outside siteTree.ts changes. Time: half a day.

Candidate 2 — site-list.tsx is higher-effort but has higher payoff for long-term maintainability. The right order is 1 → 2 → 3+4 together (natural co-location tidy), and treat Candidate 5 as a rename-and-forget. Candidate 6 resolves itself as a by-product of Candidate 3.
