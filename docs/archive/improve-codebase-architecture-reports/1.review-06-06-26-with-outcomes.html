Candidate 1

ArticleRef Propagation

Strong

Files

app/routes/article/edit/edit.tsx app/routes/article/create/create.tsx app/routes/article/site-list/site-list.tsx app/services/article.server.ts app/services/articleSiteSync.server.ts ✦ new

Problem

The ArticleRef keep-alive pattern — fetching a site record, mutating it, writing it back — was duplicated across three route actions (edit, create, site-list) in slightly different forms. Each file owned its own version of the get→transform→put loop, its own removeArticleRef/updateArticleRef calls, and its own sitesToAdd/sitesToRemove/sitesToRefresh partition logic. The deletion test: removing any one copy would have forced the others to diverge silently. The seam existed (ArticleRef mutation is a coherent concept) but there was no module behind it.

Solution

Extract into articleSiteSync.server.ts: a mutateSiteRecord atomic helper (get → transform → put with swapRecord CID), a pure computeSiteAssignmentChanges function that partitions rkey lists into add/remove/sync, and a syncSiteArticleRefs orchestrator for the two-phase edit flow. Route actions shrink to three lines.

Benefits

  • Locality: all ArticleRef mutation logic is in one file — a bug in the get/put loop is fixed once, everywhere.
  • Testability: computeSiteAssignmentChanges is a pure function; 7 tests cover add/remove/sync/rename/empty cases with no mocking.
  • Leverage: the 80-line edit action sync block became 3 lines; the seam now has a name in the domain language.

Outcome

Implemented. New file: app/services/articleSiteSync.server.ts. Test suite: app/services/articleSiteSync.test.ts (7 tests). SiteRecordValue.groups type expanded to expose slug/title so callbacks can filter without casting.

Candidate 2

Business Logic in Route Actions

Strong

Files

app/routes/article/site-list/site-list.tsx app/services/articleSiteSync.server.ts ✦ new

Problem

Three intents in site-list.tsx (deleteGroup, saveSite, removeArticle) each contained manual get→transform→put sequences. The transformation logic — filtering groups by slug, merging group/article arrays, removing article refs — was inlined directly in the action switch statement. None of it was testable without an HTTP request and a real AT Protocol agent.

Solution

The mutateSiteRecord(agent, did, siteRkey, mutate) helper introduced in Candidate 1 absorbs the get→transform→put boilerplate. Each intent becomes a pure transform callback: (record) => ({ ...record, groups: record.groups.filter(...) }). The route action becomes a thin dispatcher; the transformation is the testable unit.

Before / After

Before — logic inside HTTP handler

// deleteGroup intent (~15 lines)
const rec = await agent.com.atproto
  .repo.getRecord({ ... });
const val = rec.data.value as SiteRecord;
await agent.com.atproto.repo.putRecord({
  record: {
    ...val,
    groups: val.groups.filter(
      g => g.slug !== rkey
    ),
    updatedAt: now,
  },
  swapRecord: rec.data.cid
});

After — pure transform callback

// deleteGroup intent (3 lines)
await mutateSiteRecord(
  agent, did, siteSlug,
  (val) => ({
    ...val,
    groups: val.groups.filter(
      (g) => g.slug !== rkey
    ),
    updatedAt: now,
  })
);

Benefits

  • Locality: the CID swap and error handling are in one place; each intent only states its transform.
  • Leverage: adding a new site mutation requires one callback, not a full get/put scaffold.
  • Correctness: the swapRecord CID is always fresh — fetched inside mutateSiteRecord, not passed in stale from the loader.

Outcome

Implemented alongside Candidate 1. deleteGroup, saveSite, and removeArticle in site-list.tsx each reduced to a single mutateSiteRecord call with an inline transform. createGroup was left unchanged (conditional validation logic doesn't fit the helper cleanly).

Candidate 3

Dev Mode Fallback Duplication

Worth exploring

Files

app/routes/home/home.tsx app/routes/sites/sites.tsx app/routes/groups/groups.tsx app/routes/article/list/list.tsx app/routes/article/create/create.tsx app/routes/article/edit/edit.tsx app/routes/article/view/view.tsx app/routes/article/site-list/site-list.tsx app/routes/site/configure/configure.tsx app/services/devFixtures.server.ts ✦ new

Problem

Nine route loaders each owned their own inline dev-bypass block: a multi-line object literal returning mock data when !useRealOAuth. These blocks were untyped (or typed implicitly) against the loader's real return shape, so a shape change in the real loader would silently leave the dev fixture returning stale or missing fields. One route (edit.tsx) was silently omitting createdAt — a field the component destructured — with no TypeScript error.

Solution

Consolidate all dev fixture data into app/services/devFixtures.server.ts. One typed factory function per loader (e.g. devEditLoader(articleUrl)), with an explicit return type that mirrors the real loader's shape. TypeScript now catches shape mismatches at the fixture's definition rather than at runtime in the browser.

Before / After

Before — inline, untyped fixture

// edit.tsx loader (partial)
if (!useRealOAuth) {
  return {
    rkey: params.articleUrl,
    title: "Dev mode article",
    content: "Dev mode content",
    url: params.articleUrl,
    splashImageUrl: "",
    synopsis: "",
    // ⚠️ createdAt silently missing
    cid: "dev-cid",
    sites: [...],
    currentSiteRkeys: [],
  };
}

After — typed factory, shape verified

// devFixtures.server.ts
export function devEditLoader(
  articleUrl: string
): {
  rkey: string; title: string;
  content: string; url: string;
  splashImageUrl: string;
  synopsis: string;
  createdAt: string; // ✓ required
  cid: string;
  sites: SiteOption[];
  currentSiteRkeys: string[];
} { return { ... }; }

// edit.tsx loader (1 line)
if (!useRealOAuth)
  return devEditLoader(params.articleUrl);

Benefits

  • Correctness: explicit return types on each factory mean a real loader shape change produces a TypeScript error at the fixture, not a silent runtime mismatch.
  • Locality: all dev fixture data is in one file — easy to find, update, and keep consistent.
  • Leverage: each route loader collapses from 10–20 lines of mock data to a one-liner.

Outcome

Implemented. New file: app/services/devFixtures.server.ts with 9 typed factory functions. Pre-existing silent bug in devEditLoader (missing createdAt) caught and fixed by the explicit return type. images.tsx skipped — already had a module-level DEV_MOCK constant.

Candidate 4

Image Service Auth Replication

Worth exploring

Files

image-service/src/auth.ts app/services/auth.server.ts shared/cookieSession.ts ✦ new shared/cookieSession.test.ts ✦ new

Problem

The Image Service (port 3009, a separate Express process) needed to verify the same __session cookie that React Router signs in the main app. React Router's session storage is opaque — it doesn't export its signing primitives — so image-service/src/auth.ts contained a 69-line manual reimplementation of the HMAC-SHA256 signing format, including the non-obvious encoding detail that the JSON is btoa()-encoded before signing. If React Router ever changed this format, the Image Service would silently start returning 401s with no obvious connection to the root cause.

Solution

Extract into shared/cookieSession.ts — a single implementation of verifyScribeSession(cookieHeader, secret) that encodes the contract in one place. image-service/src/auth.ts becomes a 9-line adapter that reads SESSION_SECRET from env and delegates. The shared module is covered by 8 Vitest tests that exercise the contract directly, including the btoa encoding edge case.

Before / After

Before — two processes, duplicated contract

graph LR A[React Router\nmain app :3008] -->|signs| C[__session cookie] B[Image Service\n:3009] -->|69 LOC manual\nreplica| C C -->|used by| D[Browser] style B fill:#fee2e2,stroke:#fca5a5

After — single shared module

graph LR A[React Router\nmain app :3008] -->|signs| C[__session cookie] B[Image Service\n:3009] -->|9 LOC adapter| S[shared/\ncookieSession.ts] S -->|verifies| C C -->|used by| D[Browser] style S fill:#dcfce7,stroke:#86efac style B fill:#f0fdf4,stroke:#86efac

Benefits

  • Locality: the undocumented btoa-before-sign encoding contract lives in exactly one place with a comment explaining why.
  • Testability: 8 tests cover valid sessions, tampered signatures, wrong secrets, missing cookies, and multi-cookie headers — none of which were tested before.
  • Safety: secret is a parameter (not read from env inside the shared fn) so it's trivially injectable in tests.

Outcome

Implemented. shared/cookieSession.ts exports verifyScribeSession(cookieHeader, secret). image-service/src/auth.ts reduced from 69 to 9 lines. 8 tests in shared/cookieSession.test.ts, all passing. Both the root tsconfig.json (via **/*) and the image-service tsconfig (via relative import reference) see the shared file without config changes. CLAUDE.md updated.

Candidate 5

Type Definition Scatter

Speculative

Files

app/routes/article/list/list.tsx app/routes/home/home.tsx app/hooks/types.ts

Problem

The canonical Article type in app/hooks/types.ts (full record including content) coexists with a local type Article in list.tsx (a list-row projection with rkey/uri/cid, no content). Same name, structurally different shapes, different files. A developer importing the hooks type and the route type into the same file would face a name collision, and reading either in isolation gives no signal that a canonical version exists.

Solution

Rename local route-scoped types to names that reflect their purpose and scope, disambiguating from the canonical hooks type:

type Article → type ArticleListItem list.tsx
type RecentArticle → type RecentArticleItem home.tsx

Both types remain local to their routes — this is purely a clarity rename, not a shared-type extraction.

Benefits

  • Clarity: the Item suffix signals "view model / projection" rather than "domain record" — no name collision with the hooks type.
  • AI navigability: a search for ArticleListItem finds exactly the list-route shape; a search for Article finds the canonical hooks type.

Outcome

Implemented. type Article → type ArticleListItem in list.tsx (including the const article: Article annotation and the AssignedArticle/OrphanedArticle base type). type RecentArticle → type RecentArticleItem in home.tsx (all 3 occurrences). Typecheck clean.
