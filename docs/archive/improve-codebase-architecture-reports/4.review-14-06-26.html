Problem

buildArticleRef() is called in create.tsx and duplicated twice in edit.tsx; site assignment and CID swapping logic live inside route actions, not the persistence module.

Solution

Export createArticle() and updateArticle() from article.server.ts — each orchestrates the PDS write, ref construction, and site sync in one call. Routes become thin: validate → call → respond.

Wins

  • → Locality: ArticleRef shape changes touch one module
  • → Leverage: CID swap + site sync behind one interface
  • → Interface shrinks; route actions collapse to ~20 lines each
  • → Tests can cover create/update behaviour at a single seam
  • → Eliminates dead-code duplication in edit.tsx rename path

Decompose the Image Library route

Worth exploring in-process

app/routes/images/images.tsx (1 051 lines)

Before — one 1 051-line module

Loader (profile resolution + error fallback + dev fixtures) — calls fetch() directly
Multi-select state machine (9 useState hooks)

After — thin route + extracted modules

images.tsx — thin orchestrator (loader + action dispatch)
useImageLibrary() — multi-select + modal state + mutations
ImageGrid — pure rendering (folders + images + overlays)
imageServiceClient.ts — loader browse call moved here

Problem

At 1 051 lines, images.tsx holds every concern of the Image Library in one file — loader network calls, multi-select state, 8 modal flags, drag-drop logic, and rendering. The loader calls fetch() directly while mutations use imageServiceClient — an inconsistency that makes the seam invisible.

Solution

Extract a useImageLibrary() hook owning selection state, modal flags, and mutation calls; extract ImageGrid as a pure rendering module; move the loader browse call into imageServiceClient.

Wins

  • → Locality: multi-select bugs live in one hook
  • → Leverage: ImageGrid reusable in ImagePickerModal
  • → Interface: imageServiceClient seam covers all HTTP calls
  • → Tests can cover hook state transitions without rendering

Problem

The edit.tsx loader manually filters raw SiteRecordValue to find which sites reference an article — understanding the nested groups[].articles[] structure inline. articleSiteSync exists but exposes a callback-based mutateSiteRecord pattern that forces callers to supply a transformation that also understands the raw shape.

Solution

Give articleSiteSync.server.ts a vocabulary-level interface — findSitesContaining(uri), addToSites(), removeFromSites(), refreshInSites() — and push the raw-record knowledge into its implementation.

Wins

  • → Locality: site record structure changes stay inside one module
  • → Leverage: loader query + action mutations through one seam
  • → Interface: no SiteRecordValue in route scope
  • → Tests can verify site-finding logic without route setup

Problem

The name "ToolbarPlugin" implies it satisfies a Lexical plugin interface and could be swapped. It cannot — it is the only toolbar and it couples tightly to RichTextEditor's fullscreen state via five props. The seam is hypothetical: one adapter. The name creates false expectations for future contributors.

Solution

Rename ToolbarPlugin → EditorToolbar. Move fullscreen chrome logic (mouse timer, hide/show) fully into RichTextEditor.tsx so the toolbar receives only editor state, not view state. No interface change required — this is a naming and locality fix.

Wins

  • → Locality: fullscreen view logic lives in one place
  • → Honest name: future contributors don't build on a false seam
  • → Interface: toolbar props shrink (no more isFullscreen/chromVisible)
  • → Low risk: rename + move, no logic change

Problem

oauthClient is exported from auth.server.ts and called directly in callback.tsx and login.tsx. OAUTH_METADATA_STATIC and OAUTH_SCOPE are also exported for client-metadata.ts — the OAuth library leaks into four callers.

Solution

Wrap oauthClient calls in named functions (startOAuthFlow, handleOAuthCallback, getClientMetadata) and stop exporting the client instance. Low priority — friction is light today.

Note: The exported constants serve a genuine need — client-metadata.ts must produce JSON that matches what NodeOAuthClient was configured with. This is coupling by domain requirement, not by poor design.

Wins

  • → Locality: OAuth library swap touches one module
  • → Interface: callers need no knowledge of NodeOAuthClient
  • → Speculative — low friction today, higher if multi-account is added

Add a testable seam at route action logic

Strong in-process

app/routes/article/create/create.tsx  ·  app/routes/article/edit/edit.tsx  ·  app/routes/article/site-list/siteTree.ts (has tests)  ·  app/services/article.server.ts (has tests)

Before — complex logic, no test seam

validate fields (title, slug, content)
parse oldSiteRkeys / newSiteRkeys from FormData
buildArticleRecord()
agent.putRecord() + CID swap
slug rename: createRecord → deleteRecord
computeSiteAssignmentChanges()
removeFromSites / addToSites / refreshInSites
0 tests · all logic coupled to Request/Response

After — logic behind testable interface

article.server.ts: createArticle(fields, agent, siteRkeys)
article.server.ts: updateArticle(fields, old, agent, siteChanges)
articleSiteSync: findSitesContaining / addToSites / removeFromSites
Tests call these functions directly · no HTTP involved

Problem

The slug-rename path, CID swap, and site assignment logic in edit.tsx action carry meaningful business risk and zero test coverage. The current shape — logic embedded in a React Router action function that takes a Request — means every test path requires HTTP round-trips. This is the same root cause as candidate #1: the seam is at the HTTP boundary instead of the domain boundary.

Solution

This is the test-coverage corollary of deepening article.server.ts (candidate #1). Once logic moves there, tests can call createArticle() and updateArticle() directly with a mock agent — no React Router setup needed.

Wins

  • → Leverage: one mock agent covers all action tests
  • → Locality: slug-rename and CID-swap bugs have a home
  • → Interface: test surface matches the domain action, not HTTP
  • → Directly linked to candidate #1 — one effort, two wins

Top recommendation

Deepen article.server.ts

Candidates #1 and #6 are the same refactor viewed from two angles: move createArticle and updateArticle into article.server.ts and the three wins arrive together — duplicated buildArticleRef calls disappear, route actions shrink to orchestration only, and the business logic finally has a test seam that doesn't require HTTP. It's the highest-leverage single change in the codebase right now.

Risk: Low Scope: 2–3 files Linked: candidate #3 is a natural follow-on
