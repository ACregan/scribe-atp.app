# Extract a Site manifest module from the site-list mega-action

## Context

The `/improve-codebase-architecture` review (`docs/archive/improve-codebase-architecture-reports/5.review-05-07-26_02-13.html`) flagged `app/routes/article/site-list/site-list.tsx`'s `action` function as the sharpest shallow module in the codebase: ~610 lines (147-762), 8 `_intent` branches, each hand-rolling AT Protocol repo orchestration and manifest mutation inline. It has zero test coverage — the only tested code in this feature area is the pure helpers in `siteTree.ts`.

This plan collapses 6 of those 8 intents (`createGroup`, `deleteGroup`, `saveSite`, `removeArticle`, `publishArticle`, `moveToDraft`) into a deep server module, `app/services/siteManifest.server.ts`, following the existing house pattern set by `app/services/articleSiteSync.server.ts`. `shareToBluesky` and `notifySubscribers` are an explicit **fast-follow**, not touched here (though one small internal helper is added now to save them duplicate work later).

The workflow is characterization-first: pin the current action's behavior with tests before touching it, extract under that safety net, then add a deeper module-level suite as the long-term primary coverage.

## Four bugs to fix alongside the extraction (user-approved)

Found while verifying the action's exact behavior against the codebase. All four are one-line, low-risk fixes, applied as part of the relevant intent's extraction commit (not a separate cleanup pass):

1. **`createGroup`** has no `try/catch` around its real-OAuth PDS calls, unlike all 5 sibling intents — a PDS failure (e.g. concurrent-edit CID mismatch) throws past the action instead of returning `{ error }`. Add the same try/catch pattern used elsewhere.
2. **`moveToDraft`** can append a duplicate `ArticleRef` if invoked on an article already in `ungroupedArticles` (only prevented today by the UI hiding the button for non-grouped articles). Filter out any existing match by `uri` before appending in `ungroupedArticles`.
3. **`publishArticle`**'s `JSON.parse(siteAssignmentsRaw)` sits outside any try/catch (unlike `saveSite`'s equivalent parse) — malformed form data crashes the action. Wrap it; degrade to `{ ok: false }` on parse failure.
4. **`publishArticle`** returns `{ ok: false }` with no `error` field on failure, but the component's toast effect is `if (ok) {...} else if (data.error) {...}` (site-list.tsx:1110) — the failure branch can never fire today. Add `error: string` to the failure return so existing frontend code starts working.

## New module: `app/services/siteManifest.server.ts`

Sibling to `article.server.ts` and `articleSiteSync.server.ts`. One function per intent; `agent` is always a real, resolved `Agent` (never `Agent | null`) — dev-bypass branching (`if (!useRealOAuth) return <literal>`) stays in the route, matching existing convention, and deliberately **not** folded into the module (that's the separate, not-yet-approved "centralize `useRealOAuth`" candidate from the architecture review — out of scope here).

Every I/O function owns its own try/catch and returns a result value — no try/catch left in `site-list.tsx` after extraction; the action becomes pure dispatch.

Top-of-file comment disambiguates this module's name from the unrelated `SiteManifest` *type* already exported by `./siteTree.ts` (the loader's normalized read-side shape) — this module's functions take/return `ArticleRef`/`SiteGroup` DTOs and the raw `SiteRecordValue`, never `SiteManifest`.

```ts
// Pure — no I/O, fully unit-testable with zero mocking
export function validateGroupFields(
  title: string,
  slugInput?: string,
): { slug: string } | { error: string };

// Given old group-membership map + new tree + domain/basePath, compute which
// published-article documents need a path/canonicalUrl rewrite. Pure — the
// per-document "does it actually need to change" check stays in the I/O
// wrapper below (it needs the document's current `path`, only known after a
// getRecord per candidate).
export function computeDocumentPathUpdates(
  oldGroupByUri: Map<string, string>,
  groups: SiteGroup[],
  ungroupedArticles: ArticleRef[],
  domain: string,
  basePath: string,
): Array<{ rkey: string; newPath: string; newCanonicalUrl: string }>;

// I/O — each catches internally, returns a result, never throws
export async function createGroup(
  agent: Agent, did: string, siteSlug: string,
  fields: { title: string; slug: string },
): Promise<{ ok: true } | { error: string }>;

export async function deleteGroup(
  agent: Agent, did: string, siteSlug: string, groupSlug: string,
): Promise<{ ok: true; deletedSlug: string } | { ok: false; error: string }>;

export async function saveSiteOrder(
  agent: Agent, did: string, siteSlug: string,
  data: { groups: SiteGroup[]; ungroupedArticles: ArticleRef[] },
): Promise<{ ok: true } | { error: string }>;

export async function removeArticleFromSite(
  agent: Agent, did: string, siteSlug: string, uri: string,
): Promise<{ ok: true } | { ok: false; error: unknown }>; // route awaits, discards, still always redirects

export async function moveArticleToDraft(
  agent: Agent, did: string, siteSlug: string, uri: string,
): Promise<{ ok: true } | { ok: false; error: unknown }>; // same

export async function publishArticleToGroup(
  agent: Agent, did: string, siteSlug: string,
  params: {
    uri: string; groupSlug: string; canonicalSiteRkey: string;
    siteAssignments: Array<{ rkey: string; domain: string; basePath: string }>;
  },
): Promise<
  | { ok: true; uri: string; groupSlug: string; warning?: string;
      notification: { publicationUri: string; siteTitle: string; articleTitle: string; canonicalUrl: string } | null }
  | { ok: false; error: string } // bug fix #4: now includes error
>;

// Private, not exported — used by publishArticleToGroup now; shareToBluesky
// (fast-follow) will import it too instead of tripling the copy-paste.
async function uploadCoverImageBlob(agent: Agent, coverImageUrl: string): Promise<unknown | undefined>;
```

**`saveSiteOrder` implementation note**: replace its current manual getRecord+putRecord for the manifest overwrite with the existing `mutateSiteRecord(agent, did, siteSlug, mutateFn)` helper from `~/services/articleSiteSync.server.ts` — its callback receives exactly the old `scribe` value needed to build `oldGroupByUri`/`domain`/`basePath` before returning the new value. This de-duplicates the two different site-record-mutation patterns in the codebase down to one. No change needed to `mutateSiteRecord`'s signature (verified: no intent needs the CID/record it discards).

Route intent branches become, e.g.:

```ts
if (intent === "deleteGroup") {
  const groupSlug = formData.get("rkey") as string;
  if (!groupSlug) return { ok: false, error: "Missing group ID." };
  if (!useRealOAuth) return { ok: true, deletedSlug: groupSlug };
  const agent = await getAtpAgent(did);
  return deleteGroup(agent, did, siteSlug, groupSlug);
}
```

## Test plan (characterization → extract → deepen)

**Step 1 — before touching `site-list.tsx`:** write characterization tests importing `action` directly from the untouched file.

- Real `Request` objects (no cast): `new Request("http://localhost/x", { method: "POST", body: formData })` — precedented by `theme.server.test.ts`'s use of a real `Request` for a loader-adjacent function; confirmed Node's native `Request`/`FormData` work fine under this repo's `jsdom` vitest environment (jsdom doesn't implement `fetch`, so Node's globals pass through untouched).
- Mock the `Agent` via the existing `articleSiteSync.server.test.ts` pattern: a plain object literal cast `as unknown as Agent`, only the `com.atproto.repo.*` methods needed, each `vi.fn().mockResolvedValue(...)`.
- **Two test files**, each with a fixed, static `vi.mock("~/services/auth.server", () => ({ ... }))` — avoids any `vi.resetModules()`/dynamic-import complexity since Vitest gives each file its own module registry:
  - `site-list.action.real.test.ts` — `useRealOAuth: true`. Full matrix per intent: validation failures, success, PDS-failure. Explicitly named test cases for the current (pre-fix) behavior of the 4 bugs above, asserting the *fixed* behavior since we're fixing them in the same pass — e.g. "createGroup returns `{ error }` (not an uncaught throw) when the PDS getRecord rejects."
  - `site-list.action.devBypass.test.ts` — `useRealOAuth: false`. One cheap assertion per intent: the fixed literal is returned and `getAtpAgent`/agent methods are never called.
- Confirm both files pass against the untouched action.

**Step 2 — extract, one intent per commit, in this order** (thinnest/lowest-risk first):
1. `deleteGroup` — proves the whole harness end to end (already uses `mutateSiteRecord` unmodified).
2. `createGroup` — introduces the pure/impure split (`validateGroupFields`) and the manual getRecord→dup-check→putRecord sequence (kept manual — doesn't fit `mutateSiteRecord`'s "callback always returns a value to write" contract, since it needs an early bail-without-writing on duplicate slug). Apply bug fix #1 here.
3. `removeArticleFromSite` + `moveArticleToDraft` together — same void→result-object shape change (§ interface above). Apply bug fix #2 in `moveArticleToDraft`.
4. `saveSiteOrder` — extract `computeDocumentPathUpdates` (pure) first, then the thin I/O wrapper; switch the manifest overwrite to use `mutateSiteRecord`.
5. `publishArticleToGroup` — most complex (blob upload, two collections, multi-site fan-out via `findSitesContaining`); extract `uploadCoverImageBlob` as a private helper first. Apply bug fixes #3 and #4 here.

After each commit, re-run the Step 1 test files unchanged against the refactored action — green means the extraction (plus the one approved bug fix for that intent) didn't otherwise change behavior.

**Step 3 — add `app/services/siteManifest.server.test.ts`** as the long-term primary suite: edge cases awkward to trigger through the full action/Request harness (concurrent-edit CID mismatch, duplicate slug, `publishArticleToGroup`'s secondary-site partial failure, `computeDocumentPathUpdates` exhaustively with zero mocking).

**Step 4 — trim the Step 1 action-level tests** down to dispatch-only smoke tests (does `_intent=X` call the right module function with the right args) once Step 3 lands, so every edge case isn't double-maintained at both layers.

## Known, unchanged limitations (not regressions, not fixed here)

- `findSitesContaining` (`articleSiteSync.server.ts`) has no pagination (`limit: 100`, no cursor) — a pre-existing ceiling on `publishArticleToGroup`'s secondary-site fan-out, unrelated to this extraction.
- The 5 return-shape conventions across intents (bare `{ error }` vs `{ ok: false, error }` vs `{ ok: false }`) are preserved exactly, not harmonized — normalizing them is a frontend-coupled change and explicitly out of scope for this pass.
