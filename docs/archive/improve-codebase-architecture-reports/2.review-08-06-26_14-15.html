Deepening Opportunities

Five candidates where the codebase is carrying complexity at the wrong altitude — either scattered across callers or locked inside route actions where it can't be tested. Presented in priority order.

CANDIDATE 1

ArticleRef Propagation

The keep-alive pattern is load-bearing but fragmented across 4 files with duplicated mutation loops

Strong
services/article.server.ts routes/article/create/create.tsx routes/article/edit/edit.tsx routes/article/site-list/siteTree.ts
Problem

The logic for "keep all ArticleRefs in sync" is scattered across 4 files. To understand what happens when an article is saved, you must read edit.tsx's diffing logic (~80 lines), siteTree.ts's two mutation functions (each loops over groups + ungroupedArticles), and the service's buildArticleRef. The mutation loops in removeArticleRef and updateArticleRef are structurally identical — both walk groups and ungrouped. Adding a new ArticleRef field requires touching 5 locations (CLAUDE.md documents this burden explicitly).

Solution

Extract two pure functions from the route action into a service module: computeSiteAssignmentChanges(oldRkeys, newRkeys, slugChanged) returns a typed diff; a second function takes a SiteRecord and applies an ArticleRef mutation (add/remove/update) with one traversal. The edit.tsx action then becomes: compute diff → fetch sites → apply mutations → write back. The mutation logic has a single home and can be tested in isolation with no React Router harness.

▶ Benefits in depth

Locality: the entire "keep ArticleRefs consistent" concern lives in one file. A future developer — or a future AI — can read it once and trust it. Today you need to mentally assemble four files plus a CLAUDE.md paragraph.

Leverage: computeSiteAssignmentChanges is pure: given old and new rkey lists and a slugChanged bool, it returns a typed diff. A 20-line test file covers every case — add, remove, rename, refresh, no-op.

Tests that currently don't exist could: verify slug rename produces an update (not a remove+add), verify a no-op save still refreshes all member sites, verify that removing the last site also removes the ungrouped article ref. These are the cases most likely to silently break.

CANDIDATE 2

Business Logic Locked in Route Actions

Intent dispatch, site assignment diffing, and rkey generation are embedded in actions — tested routes cover 0% of this logic

Strong
routes/article/edit/edit.tsx routes/article/site-list/site-list.tsx routes/sites/sites.tsx routes/article/list/list.tsx
Problem

Route loaders and actions are the hardest things to unit-test in React Router — you need the full framework harness. Currently the logic for computing which sites to add/remove, generating site rkeys from domains, filtering orphaned articles, and dispatching multi-intent actions is all embedded inside these functions. The test suite has zero route tests. The logic most likely to produce a silent data corruption bug (the wrong ArticleRef updated, a site record overwritten) has no test coverage at all.

Solution

Apply a consistent extraction pattern: pure helper functions for the logic, thin action wrappers for the I/O. Concrete extractions: computeSiteAssignmentChanges() (from Candidate 1), generateSiteRkey(domain) ("norobots.blog" → "norobots-blog"), buildArticleIndex(articles, sites) for orphan detection, and intent handler functions (handleCreateGroup, handleDeleteGroup, etc.) that take agent + data and return the updated record value without performing the write.

CANDIDATE 3

Dev Mode Fallback Duplication

useRealOAuth is checked in ~9 routes; each owns its own fallback data

Worth exploring
routes/article/create/create.tsx routes/article/edit/edit.tsx routes/article/list/list.tsx routes/article/site-list/site-list.tsx routes/sites/sites.tsx routes/groups/groups.tsx routes/home/home.tsx routes/images/images.tsx
Problem

Every route that touches the PDS has its own if (!useRealOAuth) return devData block. The dev data structures are duplicated (e.g. the same site shape appears in sites.tsx, site-list.tsx, and list.tsx). When you add a new field to SiteData or change what a loader returns, you touch 8+ files. When dev mode breaks (a rare fallback returns the wrong shape), the symptom is a runtime error in development with no obvious cause.

Solution

Extract a devFixtures.server.ts module that exports typed fixture factory functions: devSites(), devArticles(), devSiteWithGroups(slug) etc. Routes check useRealOAuth once per loader and delegate to the fixture factory. All fixture data lives in one file; a type mismatch at the call site is a compile error rather than a runtime surprise. As a bonus, this file doubles as the canonical reference for "what shape does the dev environment use?"

site-list.tsx → if (!useRealOAuth) { return { site: {groups:[...], ...} } }

▶ Tradeoffs

This is a medium priority because dev mode is rarely the source of production bugs. The main payoff is compile-time safety — if a route's loader return type and the fixture don't match, TypeScript catches it at build time. The work is also mechanical: search-and-replace with a thin factory layer. Cost is low; benefit is concentrated in developer ergonomics rather than correctness.

CANDIDATE 4

Image Service Auth — Replicated Cookie Verification

The session cookie is verified by two separate implementations that must stay in sync manually

Worth exploring
app/services/auth.server.ts image-service/src/auth.ts
Problem

The main app uses React Router's createCookieSessionStorage. The Image Service has a 69-line manual HMAC-SHA256 implementation that reverse-engineers the same cookie format. The two must stay in sync. CLAUDE.md documents the encoding: "React Router serialises the session as btoa(JSON.stringify(data)).hmacSignature" — this is an undocumented internal contract with the framework. If React Router changes its cookie format, the Image Service silently starts returning 401s with no obvious cause. There is no test verifying the two agree.

Solution

Extract the cookie verification into a shared TypeScript module in a shared/ or packages/auth/ directory used by both processes. The module exports a single function: verifyScribeSession(cookieHeader, secret): string | null (returns DID or null). Both the main app (where applicable) and the Image Service import it. A single unit test verifies the encoding and HMAC logic. This is a one-seam, two-adapter situation — the seam already exists implicitly; this makes it explicit.

⚠ Practical note
Sharing code across the monorepo boundary requires a decision: a shared/ directory at the repo root with a path alias in both tsconfig.json files, or a small local packages/auth with its own package.json. The simpler option (shared dir + aliases) avoids a build step. Either way, this is a one-time, low-risk change — the Image Service auth.ts is self-contained and doesn't entangle with anything else.
CANDIDATE 5

Type Definition Scatter

Local Article/SiteData types are re-declared in individual route files, diverging from the canonical definitions

Speculative
app/hooks/types.ts app/components/types.ts routes/article/list/list.tsx routes/home/home.tsx routes/article/site-list/siteTree.ts
Problem

The canonical Article type lives in hooks/types.ts but a different Article type is locally declared in list.tsx ({ rkey, uri, title, cid, createdAt }) and a RecentArticle type in home.tsx. These are narrower projections of the same domain concept. The two SiteData shapes (component layer vs DnD tree layer) are an intentional seam documented in CLAUDE.md — less of an issue. The per-route re-declarations are the pain point: a reader has to mentally reconcile "is this the real Article or a local one?"

Solution

The projections are legitimate (routes only need a subset of fields from the PDS). The fix is naming discipline rather than structural change: rename the local types to be explicitly projections — ArticleListItem, RecentArticleItem — so it's obvious at a glance they are display-layer views of an Article, not a competing definition of the concept. hooks/types.ts remains the canonical source for the full domain types. No structural change needed; this is a rename + import-from-canonical pattern.

Top Recommendation

Start with Candidate 1 + 2 together

Candidates 1 and 2 are the same problem from two angles: business logic that belongs in a service layer is living in route actions. Candidate 1 (ArticleRef propagation) is the highest-value extraction because it has a documented test-coverage gap (CLAUDE.md explicitly lists 5 mutation sites), a clear pure-function shape, and a known failure mode (silent data corruption if a new field isn't mirrored). Doing Candidate 2 broadly at the same time (extracting intent handlers from site-list.tsx) gets you 80% of the untested surface in one pass.

Effort
~1–2 sessions
Risk
Low — pure extraction, no behaviour change
Payoff
Unit tests for the most dangerous path in the codebase
Candidates 3 and 4 are worth a conversation — 3 is mechanical (find/replace with factory), 4 requires a structural decision about the monorepo. Candidate 5 is the lowest priority: it's a naming discipline change with no structural impact.
