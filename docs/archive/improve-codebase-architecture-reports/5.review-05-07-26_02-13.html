Problem — the action is the interface: 830 lines, 6 intents, each hand-rolling repo orchestration and manifest mutation inline. Interface complexity matches implementation complexity — shallow by definition.

Solution — extract a Site manifest module (deep) with one method per intent; the action becomes a thin dispatcher that matches `_intent` to a call.

  • Leverage: 6 methods testable without a React Router action harness
  • Locality: manifest-mutation bugs concentrate in one module
  • Interface shrinks to 6 names; implementation absorbs the repo calls
  • Unblocks the "next priority" test coverage CLAUDE.md already flags

Problem — every route hand-rolls `agent.com.atproto.repo.*` and reshapes JSON itself. `article.server.ts` is a real adapter used by 2 of 5 routes; the other 3 (list, sites, configure) never reach for it — the seam exists but isn't the seam everyone uses.

Solution — widen `article.server.ts` into a repository module per collection (`site.standard.document`, `site.standard.publication`) covering list/get/create/put/delete; routes call the module, never the agent directly.

  • Two real adapters justify the seam: live agent, dev-bypass mock
  • Leverage: one interface, 5+ call sites
  • Same module absorbs candidate 5's dev/real branching (see below)
  • Foundational — candidate 1's SiteManifest module sits on top of this repository

3. Deepen the Site tree editing hooks

Worth exploring in-process

useDirtyTree.ts · useSiteListDnD.ts · siteTree.ts · site-list.tsx

Before — cross-section, thin layers

site-list.tsx — threads 13 values
useDirtyTree — tree, isDirty, markSaved
useSiteListDnD — sensors, drag handlers
siteTree.ts — pure tree<->manifest mapping
useSiteTree()
tree, isDirty, save, drag handlers — one hook

Problem — one tree edit requires reading all three files; `site-list.tsx` manually wires `useDirtyTree`'s output into `useSiteListDnD`'s input and destructures 13 values. `nodeFromRef`/`articleRefFromNode` are pure and tested, but the caller still carries all the tree-shape complexity — the extraction didn't reduce it.

Solution — merge the two hooks behind one `useSiteTree(site)` interface that internally composes drag state and dirty tracking; `siteTree.ts` stays as its internal, privately-used seam.

  • Locality: tree-edit bugs live in one hook, not three files
  • Interface drops from 13 destructured values to one object
  • Internal seam (siteTree.ts) stays testable in isolation

Problem — one component owns history, block-type, link-editing, speech-to-text and shortcut state together. Interface (props + internal state) is nearly as complex as the implementation — testing "insert a link" requires mounting the whole toolbar.

Solution — pull each concern into its own hook with a narrow interface (command + current state); `EditorToolbar` composes them and renders buttons.

  • Each command hook independently testable through its own interface
  • Locality: a link-editing bug can't hide in speech-input state
  • Toolbar shrinks to composition + rendering

Problem — a one-line boolean interface (`useRealOAuth`) still forces every one of 76 call sites to re-derive the same "mock or call the agent" branch. Interface is trivial; the decision it triggers is duplicated 76 times.

Solution — fold the branch into the repository module from candidate 2 (or `requireAtpAgent` where a repository isn't the right seam) so call sites just call the module.

  • Leverage: one branch, not 76
  • Two real adapters (live / mock) justify keeping the seam
  • Distinct from the deferred oauthClient-sealing candidate — this is about the dev/real read branch, not OAuth client construction

6. Watch components/types.ts and utils.ts

Speculative in-process

app/components/types.ts (36 lines) · app/components/utils.ts (37 lines)

Problem — types.ts already mixes Site, Article, and tree concerns (SiteCard, ArticleAssignment, TreeArticle) with no sub-grouping; utils.ts mixes URL composition with generic ID generation. Neither is painful yet at ~36 lines each, but the shape is the early stage of a misc-bucket module — an interface that grows by accretion rather than by cohesion.
Solution — no action now. If either file crosses ~100 lines or a third unrelated concern lands in it, split by concept (site-shaped exports vs article-shaped exports) rather than growing the bucket further.

Top recommendation

1 — Collapse the Site manifest action

It's the most acute shallow module in the codebase (830 lines, 6 intents, zero test coverage) and it's a concrete instance of the broader pattern in candidate 2. Extracting the Site manifest module first gives candidate 2 (the repository seam) a real second call site to design against, and directly unblocks the loader/action test coverage CLAUDE.md already flags as the next priority.
