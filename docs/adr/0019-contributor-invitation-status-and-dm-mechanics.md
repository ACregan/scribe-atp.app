# ADR 0019: Contributor Invitations Carry Status on the Roster; Invite DMs Bypass `scribe-atp-social`

## Status
Accepted and implemented — design finalized 2026-07-15 (Phase 1 grill session); full flow (invite → DM → invitee Accept → Owner-side reconciliation showing `status: "accepted"`) verified live end-to-end against two real Bluesky accounts the same day, including two bugs this verification actually caught (see Consequences)

## Context

ADR 0014/0018 modeled `scribe.contributors` as a flat, unconditional membership list — `[{did, addedAt}]` — written the instant an Owner adds someone, with no notion of consent. Working through Phase 1's concrete build (PLANNING.md) surfaced a requirement that design never accounted for: the invitee should see an explicit Accept/Reject step before they're a real Contributor, not be silently added.

This runs into the same cross-repo write asymmetry that runs through the whole feature: `scribe.contributors` lives on the Owner's `site.standard.publication` record and can only ever be written by the Owner's own session (ADR 0014's Context). An invitee's own session can never flip a field inside the Owner's record directly, so "accept" can't be a single atomic write the way it would be if the roster lived in the accepting party's own repo.

Separately, Phase 1 also needs to notify the invitee that an invitation exists at all. The only existing DM mechanism in this codebase is `scribe-atp-social`'s `/notify` route — but it was purpose-built for one thing (announcing a newly published article to a publication's *subscribers*, resolved from `action_events`) and has no way to target one arbitrary, specific DID with an arbitrary message. Extending it to do so would give it a second, structurally different responsibility that ADR 0015 already deliberately kept out of that service — anonymous, cross-site engagement events are `scribe-atp-social`'s whole reason to exist, and an authenticated-only, CMS-session-driven invitation doesn't fit that shape any better than `pending_submissions`/`contributor_memberships` did.

## Decision

**1. `scribe.contributors` entries carry a `status` field.** Shape becomes `{did, addedAt, status: "invited" | "accepted" | "rejected"}` — extending, not reversing, ADR 0018's "no *permission*-tiered role field" decision. `status` is a lifecycle marker, not a permission tier; every Contributor still has identical capabilities once accepted. The Owner's own session writes the entry immediately, with `status: "invited"`, at invite time — there is no separate pre-invite record, and no privacy concern in doing so, since `site.standard.publication` is already a fully public AT Protocol record end to end.

**2. `contributor_memberships` (ADR 0015) mirrors the same `status` field**, updated at the same three transition points (`invited` → `accepted`/`rejected`). This is the table a Contributor's own login checks to discover pending invitations without a global indexer (ADR 0015's original rationale) — without `status` mirrored locally, that check would need to re-read the actual site record per candidate row to find out what changed, defeating the point of the local index.

**3. Accept/Reject is recorded locally by the invitee's own session, not written to the Owner's AT Protocol record directly** — consistent with the cross-repo constraint above. Both outcomes are reconciled into `scribe.contributors` the next time the Owner visits that site's `/article/list/:siteSlug` page (concrete instantiation of the reconciliation pattern ADR 0014 already established for submission approval — "next login" made specific to the one page that displays roster state, rather than a vaguer global check): a locally-accepted row promotes its `scribe.contributors` entry's `status` from `"invited"` to `"accepted"` in place; a locally-rejected row removes the entry from `scribe.contributors` entirely. Without the accepted-side of this sync, an accepted invitation would sit at `status: "invited"` in the public record indefinitely — harmless for Phase 1 (no permission check reads this field yet) but a live bug waiting for Phase 3, which will need `status === "accepted"` to mean something.

**4. The invite DM is sent directly from the Owner's own CMS OAuth session**, via `chat.bsky.convo.getConvoForMembers` + `sendMessage` — the same two calls `scribe-atp-social`'s `notify.ts` already makes, just executed in-process against the Owner's own live agent instead of routing through that service's fixed `AUTHOR_HANDLE` app-password identity. This keeps `scribe-atp-social` scoped to anonymous engagement events only, per ADR 0015, and means the DM genuinely comes from the site Owner, not a generic service account.

**5. The `chat.bsky.convo.*` OAuth scope is added in Phase 1, not deferred to Phase 5.** ADR 0016 originally scoped this addition (and its mandatory one-time re-authentication for every existing user) to Team Chat. Decision 4 above needs the same scope three phases earlier. Phase 5 inherits the already-granted scope for free and does not trigger a second re-authentication event.

**Resolved during implementation:** the exact scope syntax, left unverified when this ADR was first written (see the superseded Consequences bullet below), turned out to matter — `chat.bsky.convo` is a service-proxied lexicon (proxied to `did:web:api.bsky.chat#bsky_chat`), not a repo collection, so none of this app's existing `repo:`/`blob:` scope forms apply. Per atproto.com's permission spec, service-proxied RPC methods use `rpc:<lexicon-method>?aud=<did>` — confirmed against the spec's own literal examples (`rpc:app.example.moderation.createReport?aud=*`), not the legacy `transition:chat.bsky` scope (which exists for apps that haven't migrated to fine-grained scopes at all — this app already has, via its `repo:` scopes, so mixing in a transitional scope would be inconsistent with everything else in `OAUTH_SCOPE`). `OAUTH_SCOPE` now includes `rpc:chat.bsky.convo.getConvoForMembers?aud=did:web:api.bsky.chat#bsky_chat` and `rpc:chat.bsky.convo.sendMessage?aud=did:web:api.bsky.chat#bsky_chat` (`getMessages` intentionally omitted — not needed until Phase 5's polling).

**6. The Accept/Reject prompt is a global, on-login check, not a dedicated route.** The invite DM's link carries no identifying parameter — it points at the app root. Once authenticated, a shared check (independent of which page the invitee lands on) reads `contributor_memberships` for `status: "invited"` rows against the logged-in DID and surfaces the modal. This works whether the invitee arrives via the DM link or logs in organically later, and requires no dedicated "review your invitations" route in Phase 1.

**Found live, 2026-07-15, fixed same-day:** the modal never appeared for a real invitee, despite the roster write and local `contributor_memberships` row both being confirmed correct. Root cause: reading the Owner's site record via `agent.com.atproto.repo.getRecord({repo: ownerDid, ...})` sends the request to the *invitee's own* PDS (an authenticated agent's calls target the caller's configured service endpoint), which cannot serve a record it doesn't host. Confirmed with two real accounts on genuinely different PDS hosts (`oyster.us-east` vs `rhizopogon.us-west`) — the cross-host read returned `RecordNotFound`. This is the same PDS-resolution gap already documented elsewhere in this codebase for the old (deleted) `app/hooks/` read implementation, now resurfaced because Decision 6 is the *first* genuinely cross-account read anywhere in the Contributors feature — every other call in this feature reads the caller's own repo, where this never bites. Fixed by resolving the Owner's real PDS from their DID document (same steps as `@scribe-atp/core`'s internal, not-publicly-exported `resolvePds`, reimplemented locally rather than reached across the package boundary for one lookup) and reading the record with a plain unauthenticated `fetch` against that host — no agent needed at all, since public records need no auth regardless of which PDS serves them.

## Consequences

- `scribe.contributors` and `contributor_memberships` must be kept in lock-step on every status transition (three write sites instead of one), rather than the single write ADR 0014/0015 originally assumed for "add a Contributor."
- A rejected entry is visible in the Owner's public AT Protocol record (`status: "rejected"`) for an indeterminate window, until the Owner next opens that site's article-list page. This is accepted as the same bounded, self-correcting eventual-consistency posture already applied elsewhere in this feature (ADR 0015, ADR 0017) — not a new category of risk.
- Every existing CMS user must revoke and re-authenticate before Phase 1 ships (not Phase 5), to pick up the new `chat.bsky.convo.*` scope. At the time of this decision there is exactly one user, so the cost is negligible now — but this is a one-time hard requirement for whoever deploys Phase 1, not an optional follow-up.
- ~~The exact Bluesky OAuth scope string for `chat.bsky.convo` methods is unverified~~ — resolved, see Decision 5 above. ~~What's still genuinely unverified: whether the `#`... should appear literal or pre-encoded as `%23`~~ — **resolved 2026-07-15 by a real end-to-end test.** The literal-`#` form was confirmed correct twice over: it appeared intact (not mangled or double-encoded) in Bluesky's own consent screen scope list, and — the test that actually matters — a real invite sent from one live Bluesky account arrived as a real DM on a second live account's Messages. The full chain (OAuth scope grant → `getConvoForMembers` → `sendMessage`) works against production Bluesky, not just in shape.
- **Found by that same real-account test, fixed same-day:** the invite link arrived as inert plain text, not a clickable link. `chat.bsky.convo.defs#messageInput` has no auto-linkification of bare URLs in `text` — unlike some post-rendering surfaces, Bluesky's chat UI requires an explicit `app.bsky.richtext.facet#link` facet with UTF-8 *byte* offsets into the message text. Fixed by computing that facet (`linkFacet` in `contributorRoster.server.ts`) rather than relying on the plain string. **Not fixed as part of this change:** `scribe-atp-social`'s existing `notify.ts` sends its subscriber-DM links the same unfaceted way and has the identical bug — out of scope here (different repo, pre-existing behavior, not introduced by this feature), but worth a follow-up ticket since it's now a known, fixed pattern to copy.
