# ADR 0016: Team Chat via Bluesky's `chat.bsky.convo` Lexicon, Built Inline, Not Chained Across Roster Changes

## Status
Accepted — design finalized 2026-07-15; still the current high-level design record. Renamed "Team Chat" → **Site Chat** and given concrete implementation decisions (layout, polling, pagination, error handling, scope) in ADR 0025 (2026-07-16) — read that one alongside this for anything past the architecture-level decisions here. This ADR's title retains the old name as a historical artifact.

## Context

Contributors and Owners working on the same site have a natural need to talk to each other about submissions ("I've submitted the article" / "I'll review tonight"). Rather than building a comment thread or messaging feature from scratch, the design reuses Bluesky's own chat system, `chat.bsky.convo`. This is not a novel integration for this codebase — `scribe-atp-social/src/routes/notify.ts` already calls `chat.bsky.convo.getConvoForMembers` and `sendMessage`, via the `did:web:api.bsky.chat#bsky_chat` `atproto-proxy` header, for the existing DM-notification feature. The messages this produces are real Bluesky conversations, not a shadow system — anyone in the conversation sees identical content whether they open it through the CMS or through bsky.app's own Messages section directly.

Two things had to be decided: how much of the chat experience lives inside the CMS versus handing off to Bluesky's own UI, and how to handle the fact that a site's contributor roster changes over time.

**Build vs. link out.** Everywhere else this app touches a Bluesky-native surface (viewing a cross-posted article, sharing), it links out to bsky.app rather than reimplementing Bluesky's UI. That was the initial recommendation here too — minimal new surface area, nothing to maintain long-term. The user chose to build an inline panel instead, aware of the trade-offs below.

**Roster changes and conversation identity.** `getConvoForMembers` resolves a conversation strictly by the exact set of participant DIDs — it has no topic or context field. This was actually the deciding factor behind a site's roster (ADR 0018) being scoped to exactly one site rather than one roster spanning several: if the same people discussed two different sites in one shared conversation, there would be no way to keep those two topics apart within Bluesky's own system. Even scoped to one site, though, the same mechanic causes a second problem: every time a site's roster gains or loses a member, `getConvoForMembers` for the new member set resolves to a **different** conversation than before, not the same thread with someone added or removed. There is no continuous, single "this site's chat" — there is a new conversation object every time the exact membership changes, and Bluesky's chat system has no facility to merge or link them.

## Decision

Build an inline chat panel (not a link-out) living on the per-site management page, always resolved via `getConvoForMembers(currentRoster)`.

**Do not chain or merge conversations across roster changes.** The panel always reflects whichever single conversation matches the *current* roster, in full, unmodified — never a synthetic stitched view spanning several distinct conversations from different roster eras. This was a deliberate choice over the alternative (maintaining an ordered list of conversation IDs per site and merge-rendering their histories into one continuous-looking thread): chaining is mechanically cheap, but it would let the CMS show a newly-added member history from before they joined — history they are not actually a member of on Bluesky's own systems, and would not see if they opened bsky.app's Messages section directly for the same site. That inconsistency (same person, different answer depending which surface they use) undermines the exact property that made reusing Bluesky's real chat system worthwhile in the first place — that it is the same conversation everywhere, not a shadow system the CMS curates. Not chaining preserves that property at the cost of fragmented, non-continuous history whenever the roster changes.

## Consequences

- **History fragments on every roster change.** A new contributor joining, or an existing one leaving, means the site's "current" chat is a fresh, empty conversation going forward. Prior conversations under a different membership snapshot still exist and are still real, but the CMS stops referencing them — they are not deleted, merged, or made reachable through the current view.
- **Revocation does not actually revoke.** Removing someone from a site's roster (ADR 0014/0018's Owner-controlled `scribe.contributors`) does nothing to their membership in whatever Bluesky conversation they were already part of — CMS access and Bluesky chat membership are two entirely disconnected systems. Even if Bluesky's chat lexicon exposes a way to leave a conversation, that is a per-account action only the removed person's own session can perform; there is no way for the Owner's removal action to force it on someone else's behalf. This is accepted as a known, bounded risk, using the same trust-based reasoning already applied in ADR 0014 to the sync-later publish mechanism: an ex-contributor may retain some lingering chat visibility after removal, in the same risk category as the publish-mechanism trust call, not something architected around for v1.
- **No push — polling required.** There is no websocket or persistent-connection mechanism in this stack. An open panel polls `getMessages` on an interval, using that user's own OAuth session, requiring interval choice, cleanup on unmount, and ideally pausing when the tab is unfocused. This is new operational territory for the app — its only prior chat API usage is a single fire-and-forget `sendMessage` per DM notification, nothing that polls continuously, so rate-limit behavior under sustained polling is untested.
- **A new OAuth scope is required.** Reading and sending messages via each member's own session needs a new `chat.bsky.convo.*` scope added to `OAUTH_SCOPE`. Existing users must re-authenticate before chat works for them at all — narrowly scoped, in that only chat-specific calls fail without it, not unrelated existing CMS usage — following the same re-authentication pattern already documented for prior scope additions.
- A real message list still has to be built even for a minimal version: sender resolution (DID to displayName/avatar), timestamps, own-vs-others styling, send-failure states, and pagination — more surface area than "an input and a submit button" suggests, though not a reason to avoid it.
