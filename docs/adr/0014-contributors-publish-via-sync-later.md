# ADR 0014: Contributors Publish Articles via Sync-Later, Not Content-Fork

## Status
Accepted — design finalized 2026-07-15, not yet implemented

## Context

The Contributors feature lets a site Owner grant other Bluesky accounts permission to write articles for their site. That permission is granted per-site, not through any separate "team" entity: a new `scribe.contributors: [{did, addedAt}]` array (per ADR 0018 — no permission-scoped role value needed) lives in the `scribe` extension object on the site's own `site.standard.publication` record, the same place `groups`/`domain`/etc. already live, written only by the site owner's own session. A Team was considered as its own free-standing, potentially multi-site entity and rejected: unlike Sites and Documents, which must be AT Protocol records because they're the actual portable content, team membership is pure CMS-internal permissioning with no reason to exist independently of the one site it grants access to — and `chat.bsky.convo` (ADR 0016) resolving conversations strictly by participant set made a roster spanning multiple sites actively harmful, not just unnecessary, since the same people discussing two different sites would otherwise collide into one Bluesky conversation with no way to keep the topics apart.

A Contributor's article is written to their own PDS, as their own `site.standard.document` record — AT Protocol has no shared or multi-owner records, so there is no way for the Owner to write directly into a repo they don't control, and no way for the Contributor to write into the Owner's `site.standard.publication` manifest (see ADR 0018 for who can hold Contributor permission and ADR 0015 for how each side learns anything has happened).

This means placing a Contributor's article into the Owner's site cannot be a single atomic write the way an Owner publishing their own article is (ADR 0013). Two shapes were considered for resolving this:

1. **Content-fork.** On approval, the Owner's own session creates a brand-new `site.standard.document` in their own repo, copying the Contributor's submitted title/content, crediting the Contributor via the existing document-level `contributors` byline field. This is a single ordinary write, entirely within the Owner's own repo, and has a real security property: the published copy is frozen the instant it's approved. The Contributor has no further write access to it. There is no window in which a Contributor could submit something benign, get it approved, and then quietly edit the same record into something unacceptable — there is nothing left of theirs to edit.
2. **Sync-later.** The Contributor's document stays the one canonical record, forever, on their own PDS. The Owner's manifest `ArticleRef` points at it by URI rather than owning a copy of the content. Approval and "the document says it's published" become two separate moments rather than one write, because only the Contributor's own session can ever flip their own document's `site` field — see Decision below for the mechanics this requires.

Content-fork was the only one of the two with a structural security boundary: once approved, a Contributor cannot alter what the Owner's site actually displays. Sync-later has no such boundary — the Owner's site is, indefinitely, showing whatever the Contributor's document currently says, with no re-review gate on subsequent edits.

## Decision

**Sync-later was chosen anyway.** The site Owner decides whether they trust a given Contributor; if that trust turns out to be misplaced, the Owner's existing Unpublish action (already shipped, unrelated to this feature) removes the `ArticleRef` from their manifest immediately. This is a deliberate, informed trade-off, not an oversight: a single canonical, Contributor-owned, continuously-editable article was judged more valuable than a hard security boundary that would require every future correction to go through a fresh submit-and-approve cycle.

The mechanics needed to make sync-later work:

1. **At submission**, the Contributor's own session writes `scribe.pendingPublish: { siteUri, submittedAt }` onto their own (still-loose) document. The Contributor never chooses a group at this stage — that decision belongs entirely to the Owner, made at approval time.
2. **At approval**, the Owner's own session adds an `ArticleRef` to their own manifest, the same as publishing their own article — except the `ArticleRef` snapshot must now be built from a public read of the Contributor's document (an external repo), not same-repo data. A real but bounded extension of `publishArticleToGroup`, not a redesign of it.
3. **On the Contributor's next CMS session** (their own subsequent login — nothing can force this to happen sooner), a reconciliation check runs over any of their documents still carrying `scribe.pendingPublish`:
   - A public read of the target site's manifest. If the document's URI now appears in `groups[].articles`, the submission was approved. The Contributor's own session performs one finalizing write: `site` flips to the Owner's `at://` URI (per ADR 0013's convention — `at://` means published, `https://reader.scribe-atp.app/...` means loose); `path`/`scribe.domain`/`scribe.canonicalUrl` are set to match the Owner's site and the group the Owner chose; `publishedAt` is stamped (the same convention `publishArticleToGroup` already uses for the ordinary single-owner flow, just extended to this path); `contributors` gets a `{did: ownerDid, role: "Publisher", displayName}` entry appended, guarded against being added twice, crediting whoever approved it using the same document-level byline field shipped in MR !117; `scribe.pendingPublish` is cleared.
   - If the manifest read comes back empty but a `pending_submissions` row (ADR 0015) still exists with `status: 'rejected'`, the submission was declined. The Contributor's session clears `scribe.pendingPublish` and the rejection is acknowledged.
4. **The review screen** the Owner uses to Approve or Reject is a close sibling of the existing `/article/view` route, not a reuse of it — that loader always scans the logged-in user's own `did`, whereas reviewing a submission means a public read of a specific Contributor's document by URI. Approve reuses the existing Publish-modal group picker; both Approve and Reject post a message into the site's team chat (ADR 0016) reporting the outcome.

## Consequences

- **The abuse scenario is accepted, not solved.** A Contributor can submit something benign, get it approved, and later edit the same document into something unacceptable. The only remedy is Unpublish, which is a manual, after-the-fact action, not a preventative one. A "this article's content has changed since publish" notification for Owners has been floated as a future safety net but is explicitly backlogged, not part of this decision.
- **If a Contributor never logs back in after approval**, their document sits in an inconsistent state indefinitely — referenced by the Owner's manifest (which is all that matters for readers, since consumer sites resolve articles via the manifest/`ArticleRef`, not the document's own `site` field) but still self-labeled loose. This is a cosmetic inconsistency for any third-party tool that scans the Contributor's repo directly, and means the Contributor's own `/article/list` keeps showing the article as "Standalone" until they return. Accepted as self-correcting.
- **If a Contributor becomes permanently unreachable and their published article needs correction**, there is no automated recourse — the Owner cannot write to a repo they don't own. The manual escape hatch is for the Owner to read the Contributor's document, write a fresh copy via the ordinary create-and-publish flow, and credit the original author via the existing `contributors` byline UI, exactly as if the article had come through the normal flow. This requires no new engineering; it is a deliberate, rare, human-driven workaround, not a designed feature.
- **Reversing this decision later (moving to content-fork) would not be a config change.** By the time that question comes up, real articles will exist in sync-later's shape — Contributor-owned documents referenced by other people's manifests — and switching mechanisms means migrating live, cross-repo content, not flipping a flag.
