# ADR 0021: Unified Publish/Submit Modal — URI Value Scheme, Grouped Select, Shared Site-Resolution

## Status
Accepted — design finalized 2026-07-16 (Phase 3 sub-pass 1 grill session), not yet implemented

## Context

ADR 0014's Decision section specs the unified Publish/Submit modal at a conceptual level — an `<optgroup>`-grouped Site dropdown (Owned Sites vs. Contributor Sites), an inline confirmation box replacing the Group dropdown for a Contributor site, the confirm button's label flipping between "Publish" and "Submit for Review". Working through the concrete build against the actual code (`list.tsx`'s existing Publish modal, `Select.tsx`, `contributorRoster.server.ts`'s already-shipped `listPendingInvitations`) surfaced several implementation-level decisions this phase is the first to actually need.

## Decision

**1. Every Site dropdown option uses the full `at://` site URI as its value — owned and Contributor sites alike** — not the bare `rkey` the existing Publish modal uses today (valid only because every option was implicitly the caller's own repo). A bare rkey is ambiguous once options can come from a different account. Concretely: `at://{ownerDid}/site.standard.publication/{rkey}`. This lets the submit action determine "publish vs. submit" for itself, by parsing the owner DID out of the submitted URI and comparing to the caller's own `did` — it never has to trust a client-supplied "which branch" flag, since the answer is self-evident from the value alone.

**2. `Select` (`app/components/Select/Select.tsx`) gains a new, additive `groups` prop, not a change to the existing `options` prop.** `options: SelectOption[]` (today's flat form) is untouched — every existing call site (Umami's connect form, role pickers, etc.) needs no changes. A caller passes *either* `options` (unchanged) *or* a new `groups: SelectOptionGroup[]` (`{ label: string; options: SelectOption[] }[]`), rendering `<optgroup>` sections — never both. Only the Publish/Submit modal uses the new form.

**3. The "Contributor Sites" list is resolved by a new function sharing code with the already-shipped `listPendingInvitations`.** Both need the identical shape of work — read `contributor_memberships` for a DID, filter by one status, resolve each surviving row's site info via the existing cross-repo-read primitives (`parseSiteUri` → `resolveOwnerPdsUrl` → plain `fetch`, all already built in `contributorRoster.server.ts`) — differing only in which status they filter for (`"invited"` vs `"accepted"`). Extracted into a shared internal helper (`resolveMembershipSites`) rather than duplicating the `Promise.allSettled`/fetch/logging block a second time.

**4. The Owner's display name (for the confirmation box's "requires approval from {Owner}" copy) is resolved eagerly, bundled into the same per-membership resolution step** — one more `fetchBskyProfile(ownerDid)` call alongside the site-record fetch already happening in the helper from Decision 3, not a separate lazy fetch triggered only once a Contributor site is selected in the modal. Chosen over lazy resolution because the Contributor Sites list for any one person is realistically small (consistent with this feature's no-global-indexer, all-pairwise design throughout), so resolving every candidate's owner profile up front costs little and avoids a network delay/spinner when the user actually picks one.

**5. The submit action enforces two guards server-side, not just client-side UI hiding:** reject submitting a document that's already published (its `site` field is an `at://` URI, not a loose reader URL), and reject submitting a document that already has `scribe.pendingPublish` set (one pending submission at a time, no overwriting/resubmitting elsewhere until the existing one resolves). Consistent with ADR 0013's "a document belongs to at most one site, ever" — pending-submission is treated as a third, transient state between loose and published, not something that can fork or be abandoned mid-flight by simply submitting elsewhere.

**6. `/article/list` shows a "Pending Review" pill in place of the Publish button for any standalone article currently carrying `scribe.pendingPublish`.** Without this, a Contributor would see a live, clickable Publish button that the server-side guard from Decision 5 would just reject — technically correct but an avoidable dead-end click. The list loader already reads the full document record per standalone article, so reading `scribe.pendingPublish` off it is a cheap addition, not a new fetch.

**7. Both branches (publish an owned site, submit to a Contributor site) share one action intent, renamed from `publishArticle` to `publishOrSubmitArticle`** for clarity — not two separate intents (`publishArticle` / `submitArticle`) the client would have to choose correctly between. The action re-derives which branch applies itself (Decision 1), so a single intent name is both simpler for the client and doesn't rely on the client's own claim about which branch it's exercising.

## Consequences

- `PublishTargetSite`'s `rkey`-keyed shape (`list.tsx`) needs to carry (or be replaced by) the full site URI: existing owned-site logic that currently keys off `rkey` needs updating to the URI-based scheme from Decision 1, even though nothing about owned-site publish *behavior* changes.
- Resolving the Contributor Sites list now does two fetches per membership (site record + owner profile) instead of one — bounded by the same small-N assumption as ADR 0015/0019's discovery mechanisms throughout this feature; would need revisiting if a Contributor ever has a large number of memberships, which nothing in this feature's design anticipates.
- **Backlogged, not specced now:** a Contributor rescinding their own pending submission before the Owner responds. Raised during this grill session as a real, missing capability, but not part of ADR 0014/0015's original design and explicitly deferred rather than designed under this ADR. Needs its own design pass later (at minimum: what happens to the local `pending_submissions` row and the Owner-side "New Article Submission" section on rescind, and whether the Owner needs to be notified that a submission was withdrawn).
