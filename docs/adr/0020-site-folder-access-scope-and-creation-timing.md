# ADR 0020: Site-Owned Image Folders Are Fully Restricted, Created With the Site, and Gated on Accepted Status

## Status
Accepted — design finalized 2026-07-15 (Phase 2 grill session), not yet implemented

## Context

ADR 0017 established the shape of site-owned Image Library folders — a nullable `site_uri` owner column on `image_folders`, a `site_rosters` local table in the Image Service's own SQLite, and a CMS-pushed sync rather than the Image Service resolving AT Protocol itself. What it left unspecified — because Phase 1 (the roster itself) didn't exist yet — is the concrete behavior once a real roster exists to sync: exactly how open or restricted a site folder is relative to personal folders, when it's created, when someone actually gains access to it, and what the sync endpoint's contract looks like. Working through Phase 2's build against the actual Image Service code (mapped fresh for this session — `image-service/src/browse.ts`, `folders.ts`, `deleteImage.ts`, `bulkOperations.ts`) surfaced five concrete gaps ADR 0017 didn't address.

## Decision

**1. Site-owned folders get real read *and* write restriction — owner or roster member only. Personal folders explicitly keep today's open-read behavior.** This is a deliberate asymmetry, not an oversight: ADR 0017's own motivating problem (unrestricted read access) is exactly what a site folder must not repeat, while personal-folder read access is unaffected and remains a separately tracked, open gap (per ADR 0017's own scope line). Concretely: `browse.ts`'s top-level "all root folders" listing must exclude a site-owned folder unless the caller is authorized for it, and fetching a specific `folderId` — which today does *no* ownership comparison at all for any folder — needs a check added.

**2. Contributors get full write parity with the Owner inside a site's folder** — upload, delete, create subfolders, move/reorganize, all covered by one "owner-or-roster-member" check. No finer-grained tier (e.g. "can upload but not delete") — consistent with ADR 0018's decision that Contributors have no tiers at all.

**3. Access is granted only once a Contributor's status is `"accepted"`, not from the moment they're `"invited"`.** This is the same reasoning ADR 0019 introduced the invited/accepted distinction for in the first place — an invitation with no response yet is not real membership. Concretely, `site_rosters` is synced from **two** places, not the one ADR 0017/PLANNING.md originally sketched: `removeContributor` (revoke immediately, regardless of prior status) and the accepted-promotion branch of `reconcileContributorStatuses` (grant access at the moment acceptance is actually reconciled). The rejected-promotion branch needs no sync call — a rejected invitee never had access to begin with.

**4. The sync endpoint is `PUT /api/image-service/site-roster`, body `{ siteUri, siteName, memberDids }`, wholesale replace** (matches ADR 0017's "delete-and-reinsert," not a diff). Authorization: the Image Service parses the owner DID out of `siteUri` and requires the session's own `userDid` to match it, 403 otherwise — only the site Owner's own session can ever push a roster update for their site, same posture as every other roster write in this feature.

**5. The sync call is made from inside `contributorRoster.server.ts`, not the route layer.** `removeContributor` and `reconcileContributorStatuses` each gain a `cookieHeader: string` parameter (threaded from `site-list.tsx`'s `request.headers.get("Cookie")`) and call the new `syncSiteRoster` (in `imageServiceClient.server.ts`, alongside the existing `browseImages`) as an internal side-effect after successfully mutating `scribe.contributors` — mirroring exactly how `inviteContributor` already calls `sendInviteDm` as a side-effect of its own write. Keeps "the roster changed, sync what depends on it" cohesive in one place rather than split across the route and the service module.

**6. The site's shared folder is created at site-creation time, not lazily on first accepted Contributor** — reversed from this session's initial recommendation. The folder is useful to the Owner as "this site's images" independent of whether any Contributor ever exists, not merely a sharing mechanism, so tying its existence to a Contributor's acceptance was too narrow a framing. Mechanically this needs no new endpoint: `sites.tsx`'s create action calls the same `syncSiteRoster(siteUri, siteName, [])` (empty roster) immediately after writing the site record, reusing the auto-create-if-missing behavior the sync endpoint already needs for its normal case. If the Image Service is unreachable at site-creation time, site creation still succeeds — best-effort, logged, self-correcting the next time anything else calls `syncSiteRoster` for that site (same fire-and-forget posture as the invite DM). **Requires a one-shot backfill script** calling this same endpoint once per existing site, since the 3 sites already live across 2 accounts predate this feature entirely.

**7. The folder's display name is `"{domain} Images"`** (e.g. "anthonycregan.co.uk Images"), using the site's `scribe.domain` rather than its title — more concise, and passed in the sync payload's `siteName` field since the Image Service never resolves AT Protocol records itself (ADR 0017). Set once at folder-creation time only — **explicitly not kept in sync with a later domain change** via `/site/:siteName/configure`. Accepted as a cosmetic, self-correcting-only-on-next-full-resync limitation, not solved now.

**8. A new shared module, `image-service/src/access.ts`, centralizes the owner-or-roster-member check.** Today's per-file idiom (`!row || row.user_did !== did`, repeated across `folders.ts` ×3, `deleteImage.ts`, `bulkOperations.ts` ×2 plus a destination-folder check) would otherwise need the same new "OR site owner OR roster member" branch copied into 7+ call sites across 4 files. One shared helper (e.g. `canAccessFolder(did, folder)` / `canAccessImage(did, image)`) replaces all of them and is the only thing `browse.ts`'s new read-side checks need too.

## Consequences

- `image_folders.user_did` must become nullable (today `NOT NULL`) alongside the new nullable `site_uri` — a folder is owned by exactly one of the two, never both, never neither. SQLite has no direct `ALTER COLUMN ... DROP NOT NULL`; this needs the standard create-new-table/copy/drop/rename procedure, not a simple `ALTER TABLE ADD COLUMN`. Low-risk at current data volume (a handful of folders across 3 sites, 2 accounts).
- `site_rosters (site_uri, member_did)` needs a composite uniqueness constraint so repeated wholesale-replace syncs don't accumulate duplicate rows.
- **Backlogged, not solved:** what happens to a site's shared folder when the site itself is deleted. Nothing in this decision or ADR 0017 addresses cleanup — the folder would be orphaned (harmless, but not cleaned up). Flagged explicitly as a deliberate omission for a future pass, not an oversight.
- The one-shot backfill script is a new one-time devtool, following the existing precedent for this kind of thing in this app (dev-only, ADMIN_DID-gated tools already exist for other one-off migrations).
- Every read endpoint in the Image Service (`browse.ts`) needs an access check added where today there is none at all for *any* folder — previously this was uniformly open by design (ADR 0017 leaves personal folders that way deliberately); Phase 2 introduces the first read-side restriction this service has ever had, scoped only to site-owned folders.
