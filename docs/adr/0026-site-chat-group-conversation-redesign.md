# ADR 0026: Site Chat — Group Conversation Redesign

## Status
Accepted — 2026-07-17, superseding ADR 0025 Decision 2 and part of ADR 0016's Consequences

## Context

ADR 0025 built Site Chat's resolution mechanism on `chat.bsky.convo.getConvoForMembers`, resolving a conversation from the current roster (Owner + every accepted Contributor) on every mount. This shipped, was verified live between two real accounts, and immediately broke: `getConvoForMembers` is explicitly documented as 1-1 only — "Always returns the same direct (non-group) conversation. To create a group conversation, use createGroup." A first attempt also had a self-inclusion bug (the caller's own DID was wrongly left in the `members` list; fixed separately), but fixing that only exposed the deeper problem — passing 2+ *other* members either errors or silently resolves to an inconsistent conversation, since the endpoint has no group semantics at all.

The user's own framing settled the scope immediately: **"this was always going to be 1 Owner + n Users. There could be 20 contributors to this site."** This was never an edge case to defer — it's the actual, current requirement, and ADR 0025's mechanism cannot support it at any scale beyond a single Contributor.

A separate, distinct lexicon namespace — `chat.bsky.group.*` — supports real, persistent, member-mutable group conversations: `createGroup`, `addMembers`, `removeMembers` (plus a join-link subsystem, unused here). Key facts from that surface:

- **`createGroup` is explicitly non-idempotent** — "It will create new groups even if the membership is identical to pre-existing groups." Unlike `getConvoForMembers`, there is no get-or-create-by-membership semantic; the resulting `convo.id` must be persisted by the caller, or every subsequent visit creates a duplicate group.
- **`createGroup`'s `members` excludes the caller** — "Will create 'pending' membership for all members, except the owner who is 'accepted'." The owner (whoever calls it) is implicitly a member; only the named `members` start in "pending"/"request" status. Same convention as `getConvoForMembers`.
- **`addMembers`/`removeMembers` operate on an existing `convoId`** and mutate real membership going forward — no re-creation needed on roster change, which directly resolves ADR 0016's own accepted "history fragments on every roster change" trade-off (a persistent group's history never resets).
- **New members added via `addMembers` start in "request" status** on the group side and must individually call `chat.bsky.convo.acceptConvo` before they can read or send. `chat.bsky.convo.getConvo({convoId})` returns `status: "request" | "accepted"` (a plain string, not a nested object) scoped to the calling viewer — this is how a session detects whether it still needs to accept.

The user also settled the remaining open question directly: **"The chat feature has no reason to exist until a contributor has been added to the site. When a contributor is added then the chat can be created."** The group is created eagerly, at the exact moment the first Contributor's status is promoted to `accepted` — not lazily on first chat visit, and not before any Contributor exists at all.

## Decision

1. **Group creation/membership sync is triggered from `reconcileContributorStatuses`, not from any chat-specific code path.** This function already runs on every Owner page-load and already computes exactly which DIDs are newly promoted to `accepted` in a given pass (previously only to write `scribe.contributors`). It now also calls a new `syncSiteChatGroup(agent, siteUri, siteName, newlyPromotedDids)`: if no group exists yet for this `siteUri`, `createGroup({ members: newlyPromotedDids, name: siteName })` and persist the returned `convo.id`; if one already exists, `addMembers({ convoId, members: newlyPromotedDids })` instead. `removeContributor` and the rejected-row branch of `reconcileContributorStatuses` symmetrically call `removeSiteChatMember`, which calls `chat.bsky.group.removeMembers` if a group has been created. All of this is best-effort (matches `sendInviteDm`'s existing posture) — a chat-sync failure never turns an already-successful roster write into a reported failure. There is no drift-reconciliation pass; a failed sync leaves that one Contributor durably missing from the group until the next roster change touches the site. Acceptable for now, not yet a problem with no real user base.

2. **The persisted mapping lives in a new local table, `site_chat_convos (site_uri PRIMARY KEY, convo_id, created_at)`.** One row per site, written once at first-group-creation and never updated — a group, once created, exists for the lifetime of the site.

3. **Chat resolution is now a local lookup, not a per-request `getConvoForMembers` call.** The group's existence and membership are entirely a function of the roster, maintained out-of-band by (1) above — not of who's currently looking at the chat panel. The `article/site-chat/:siteSlug` resource route's resolve mode now takes `?ownerDid=` (site identity), looks up `site_chat_convos` for `at://{ownerDid}/site.standard.publication/{siteSlug}`, and if found, calls `chat.bsky.convo.getConvo({convoId})` to check the caller's own membership status — transparently calling `acceptConvo` if it's still `"request"`. No Contributor ever sees a manual "accept this chat" step. If no row exists yet, the lookup returns `errorType: "notCreatedYet"`.

4. **The four `getConvoForMembers`-specific social-graph error types are retired.** `blocked`/`messagesDisabled`/`notFollowed`/`accountSuspended` were per-request 1-1 resolution failures; they no longer apply, since the group's membership was already established once, on the Owner's own session, at accept time — not re-negotiated on every viewer's lookup. The resolve error surface shrinks to `notCreatedYet` (no group exists — either the panel shouldn't be rendering at all, or a genuine race) and `unknown` (a real lookup failure, logged server-side).

5. **The Site Chat panel is hidden entirely for the Owner when there are zero accepted Contributors**, rather than rendered with a permanent "waiting" state. Per the user's framing, the feature has no reason to exist yet — there's nothing to create and nothing to show. `/article/list/:siteSlug` computes `hasAcceptedContributors` from the loader's roster and only renders the two-column layout (main content at `span={8}`, chat at `span={4}`) when the Owner has at least one; otherwise the main content takes the full `span={12}`. A Contributor viewing this page is necessarily themselves an accepted Contributor (ADR 0025's read-only access check already gates that), so the group is guaranteed to exist by the time they can see the page at all — the hide logic is Owner-side only.

6. **`useSiteChat` no longer takes or computes a member-DID list at all.** Resolution now only needs `(siteSlug, ownerDid)` — the site's own identity — since membership is no longer something the client assembles and sends. This also removes the "resolve once per roster change" mechanism from ADR 0025 Decision 2 entirely: there's nothing roster-shaped to re-resolve on, since a mounted panel's `siteSlug`/`ownerDid` don't change mid-session.

## Consequences

- ADR 0025 Decision 2 ("resolved fresh... via `getConvoForMembers`... resolve-once-per-mount") is superseded by Decisions 1–3 above. ADR 0025 Decisions 1 (layout), 3 (polling), 4 (no pagination), 6 (send-failure UX) are unaffected — `getMessages`/`sendMessage` operate identically on a group `convoId` as a direct one.
- ADR 0016's "history fragments on every roster change" trade-off is resolved, not just accepted, by this redesign — a persistent group's history survives roster changes indefinitely.
- ADR 0016's "revocation doesn't actually revoke chat access" trade-off is also improved: `removeMembers` is now a real membership removal (`InvalidConvo`/`InsufficientRole` aside), not merely a client-side conversation no longer being resolved while the underlying 1-1 convo technically still existed.
- New group-specific failure modes (`UserForbidsGroups`, `MemberLimitReached`, `InsufficientRole`, `ConvoLocked`) are not surfaced to the Owner at all — they're logged and swallowed at sync time, same posture as the existing invite-DM failure handling. Matches the user's explicit "we are our only users at the moment" risk tolerance from the Phase 5 grill session; revisit before onboarding real external Owners.
- `chat.bsky.group.*` is documented as "under active development and should be considered unstable" in its own lexicon description — a real, if currently accepted, risk that this feature depends on an unstable Bluesky API surface.
- Group creation/membership sync could not be verified against two real Bluesky accounts as part of this change (unlike ADR 0025's original 1-1 mechanism, which was live-tested) — verified via typecheck, unit tests (including the new `reconcileContributorStatuses`/`removeContributor` chat-sync assertions), and the full e2e suite only. Live verification between two real accounts with a genuine 2+-Contributor roster is still outstanding.
